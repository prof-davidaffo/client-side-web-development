Chapter 5 HTML Semantico

Come spiegato nei capitoli precedenti, lo scopo del markup HTML e’ annotare semanticamente il testo. Quando diciamo che HTML e’ “semantico”, intendiamo che descrive il significato, la struttura o lo scopo del testo — non l’aspetto visivo del contenuto. Usare HTML solo per la semantica aiuta a seguire il principio di separation of concerns (rendendo il codice piu’ flessibile e facile da modificare e mantenere). Inoltre rende le web page piu’ accessibili, perche’ possono essere comprese e navigate meglio da tecnologie assistive come gli screen reader.

Un HTML non semantico e’ un HTML che non descrive davvero il significato di un contenuto, ma che puo’ avere un aspetto visivo che suggerisce quel significato attraverso l’uso di CSS. Per esempio, un elemento <h1> indica un top-level heading, come il titolo della pagina. Ma di default i browser danno a <h1> un aspetto diverso dal contenuto non marcato (di solito piu’ grande e in grassetto). E’ possibile ottenere un effetto visivo simile solo con CSS:

<!-- HTML -->
<h1>Questo e' un vero heading!</h1>
<p class="fake-header">
  Questo solo SEMBRA un heading.
</p>
/* CSS */
.fake-header {
  font-size: 2em;
  font-weight: bold;
}
A real heading (top) and a fake heading (bottom).

In questo esempio, la seconda riga sembra un heading (e’ grande e bold) — ma l’HTML dice che non lo e’, e’ solo un paragraph. Questo puo’ creare problemi e confusione, soprattutto per screen reader e altri sistemi che non sono interessati all’aspetto visivo. Un utente cieco che visita questa pagina non saprebbe che il secondo paragraph dovrebbe fare da heading, perche’ questo non e’ indicato nell’HTML (solo dall’aspetto renderizzato).

Per questo motivo e’ vitale che il markup HTML sia usato solo per fornire un’annotazione semantica accurata. Non usare un tipo di elemento non semantico (come un heading) per il suo aspetto di default, e usa elementi che forniscono informazioni aggiuntive sullo scopo del contenuto quando possibile, cosi’ che screen reader e altre tecnologie assistive lo comprendano.

Molte regole nella code style guide riguardano l’uso semantico di HTML.

Questo capitolo discute alcuni elementi HTML specifici, fornendo dettagli su come includerli nei documenti e come usarli in modo semantico per garantire che la pagina supporti al meglio le tecnologie assistive. Discute anche lo standard ARIA per aggiungere significati semantici aggiuntivi agli elementi e supportare ulteriormente l’accessibilita’.

5.1 Elementi HTML specifici

Ci sono oltre 100 elementi HTML diversi, ognuno con il proprio significato semantico. Questo capitolo non e’ una reference completa (per quello usa la reference MDN), ma fornisce indicazioni specifiche e requisiti per usarli in modo efficace e semantico.

Hyperlink

Gli hyperlink sono fondamentali per HTML (e’ proprio nel nome!), quindi l’anchor element <a> e’ forse il piu’ importante. L’elemento <a> indica un hyperlink: testo che, quando attivato (ad es. cliccato o selezionato), fara’ navigare il browser verso un URL diverso da quello corrente. Cliccare un hyperlink si chiama seguire il link.

Come discusso in HTML Fundamentals, l’elemento <a> ha quasi sempre un attribute href che indica dove va il testo dell’elemento:

<!-- a external hyperlink to `https://ischool.uw.edu` -->
<a href="https://ischool.uw.edu">iSchool homepage</a>

Il valore dell’attribute href puo’ essere qualsiasi URL valido (inclusi quelli che usano protocolli diversi da https). La maggior parte degli hyperlink e’ di uno di questi due tipi:

  1. external hyperlink: navigano verso un domain diverso. Per esempio, un link a Google o alla homepage dell’iSchool (come l’esempio sopra) da questo libro sarebbe un external hyperlink, perche’ le pagine del libro non sono sullo stesso domain. Gli external hyperlink si scrivono in genere come URL completi, includendo il protocollo, perche’ includono il domain a cui si collegano.

  2. internal hyperlink: navigano verso altre web page nello stesso domain. Per esempio, i link ad altri capitoli di questo libro sono internal hyperlink. Gli URL per questi hyperlink sono specificati solo come path degli altri URL; il domain e il protocollo sono sottintesi e quindi omessi come ridondanti. Per esempio:

    <!-- An internal hyperlink to a different `.html` file in the same domain -->
<a href="/client-side-development.html">Chapter 2: Client-Side Development</a>

    Gli internal hyperlink possono usare absolute path come nell’esempio sopra. Il path e’ allora determinato dalla “root folder” del server. Questi URL risolvono anche se il link e’ in una pagina nidificata dentro una o piu’ cartelle del server. E’ anche possibile usare relative path, nel qual caso il target dell’hyperlink e’ relativo all’URL della pagina corrente (che di solito renderizza il file che include il link):

    <!-- An internal hyperlink to a file in a different folder on the same domain.
     Nota che l'URL relativo puo' includere la risalita di una cartella. -->
<a href="../assignments/exercise-01.html">Exercise 01</a>

    E come notato in HTML Fundamentals, l’URL di un hyperlink puo’ includere anche il fragment (la parte dopo #) per creare un bookmark link verso un elemento con un attribute id su una pagina:

    <a href="index.html#nav">Link to element on `index.html` with `id="nav"`</a>
<a href="#title">Link to element on current page with `id="title"`</a>

Per fare in modo che un hyperlink si apra in una nuova tab o finestra (dipende dal browser), aggiungi un attribute target="_blank". Il valore "_blank" e’ una keyword speciale che dice al browser di aprire il link in un nuovo contesto.

Infine, e’ importante notare che attivare un hyperlink (ad es. cliccarlo) invia una nuova HTTP request al server, che risponde con la nuova pagina da renderizzare. Questo significa che anche se hai un hyperlink alla stessa pagina (eccetto i bookmark link), la pagina verra’ comunque ricaricata:

<!-- in index.html -->

<a href="index.html">A link to this same URL will cause the page to reload</a>

Ricaricare una pagina significa che tutti gli asset (immagini, ecc.) possono essere riscaricati e qualsiasi stato in JavaScript verra’ perso. Fai attenzione a quando inneschi un reload di pagina.

HTTP e’ un protocollo stateless. Questo significa che ogni HTTP request — ogni page load — e’ considerata indipendente e senza relazione intrinseca. Visitare o caricare una pagina una volta non ha impatto su page load futuri; ogni visita e’ una “prima e ultima volta”. Server e browser possono (e spesso lo fanno) includere logica extra per tracciare sessioni e far sembrare che i page load siano correlati (ad es. mantenere un carrello online), ma questo e’ lavoro extra separato dall’encoding della pagina. Nello sviluppo client-side e’ meglio agire come se ogni page load fosse un evento unico, salvo indicazioni diverse dal server.

Gli hyperlink servono per la navigazione — attivare un hyperlink dirige l’utente verso un URL diverso. L’elemento <a> non va usato per elementi cliccabili che non fanno navigazione (come inviare un form o espandere un dropdown). I clic non di navigazione vanno annotati come <button>. Allo stesso modo, non usare un <button> per la navigazione — quello deve essere un <a>! Se vuoi che un hyperlink sembri un button, puoi usare CSS per stilizzarlo in quel modo — Bootstrap supporta facilmente questo comportamento, per esempio.

Hyperlink accessibili

Tecnologie assistive come gli screen reader possono saltare rapidamente tra gli hyperlink di una pagina — questo aiuta gli utenti a capire quali elementi di navigazione sono disponibili e a spostarsi velocemente senza ascoltare contenuti extra. Per questo, per rendere un hyperlink accessibile, il testo del link deve indicare dove porta anche quando e’ fuori contesto:

<!-- Do this -->
<a href="/assignments">see assignments here</a>

<!-- Don't do this -->
see assignments <a href="/assignments">here</a>

Nel secondo esempio, il link consiste solo nella parola “here”, quindi quando uno screen reader elenca tutti gli elementi di navigazione, elenchera’ solo la parola “here” senza contesto per capire dove porta. Questo aiuta anche utenti vedenti, che capiscono piu’ facilmente i link nel contesto.

Allo stesso modo, assicurati che testi di link “simili” che puntano a posti diversi siano resi univoci:

<!-- Don't do this; screenreaders can't distinguish between the links -->
For the first <a href="exercise-01.html">exercise</a> you'll practice HTML, for
the second <a href="exercise-02.html">exercise</a> you'll practice CSS.

Usare attribute ARIA come aria-label (vedi sotto) puo’ aiutare a garantire che il testo dei link sia accessibile, fornendo contesto specifico per lo screen reader se fai fatica a rendere la prosa auto-contenuta.

Elementi testuali inline

//storicamente le singole parole di una frase possono essere formattate tipograficamente per essere marcate come distinte rendendole in bold o italico. Ma queste sono solo proprieta’ di aspetto/formattazione — non forniscono alcuna informazione semantica sul testo. Quindi anche se HTML include elementi <b> per il bold e <i> per l’italic, questi sono artefatti storici e non dovrebbero essere usati. Per esempio, <b> fa renderizzare il testo in bold, ma non fa nulla per indicare perche’ quel testo e’ in bold. Un utente vedente puo’ inferire qualcosa dal contesto, ma uno screen reader non puo’ fare queste inferenze da elementi non semantici.

HTML include invece una vasta serie di inline textual elements che puoi usare per separare semanticamente il testo. Di default possono renderizzare il testo in bold o italico (e questo puo’ essere personalizzato con CSS), fornendo anche supporto semantico aggiuntivo per rendere il testo piu’ accessibile alle tecnologie assistive.

Alcuni degli elementi inline semantici piu’ utili o interessanti sono descritti qui sotto, con link alla documentazione MDN per maggiori informazioni:

  • <em> marca il testo come emphasized, di default in italic.

  • <strong> marca il testo come strongly importante, di default in bold. Questo e’ distinto da <em>: non e’ solo enfasi ma particolare importanza.

  • <code> marca testo come codice, di default in monospace. L’elemento <pre> puo’ essere usato per gestire testo pre-formattato per code block piu’ grandi.

  • <del> marca testo come eliminato, di default con strike-through.

  • <u> marca testo con una annotazione non testuale, come un underline o una decorazione. Anche se di default rende il testo sottolineato, il significato dell’elemento e’ annotare che quel testo ha una decorazione (anche se e’ fornita dal tag). In generale e’ meglio evitare questo elemento perche’ poco semantico (come evitare <i> e <b>), e usare la property CSS text-decoration per sottolineature.

  • <cite> marca testo come titolo di un’opera creativa citata — ad esempio il titolo di un libro o film. Nota che non indica la “citazione” in un saggio (non e’ il (Brown, 1999) in MLA, o una voce in bibliografia); indica solo il titolo dell’opera. Di default rende il testo in italic.

  • <abbr> marca testo come abbreviazione o acronimo. In particolare, se assegni all’elemento un attribute title con il significato esteso dell’acronimo, l’utente puo’ passare con il mouse e vedere un pop-up con il significato:

    <abbr title="Hypertext Markup Language">HTML</abbr>

    Questo aiuta a fornire definizioni fuori dal flusso del testo — ma quando presenti informazioni, e’ meglio includere quelle definizioni nella frase stessa.

  • <time> rappresenta un orario o una data. Fornire l’attribute datetime puo’ codificare l’orario in formato leggibile dalle macchine, utile per SEO o altri sistemi:

    <time datetime="2023-12-23">New Year's Eve</time>

Importante: anche se molti di questi elementi forniscono uno styling di default, dovresti usarli solo per scopi semantici. Non usare <em> solo per ottenere l’italic: puoi usare un altro elemento semantico (o uno <span>, vedi sotto) e poi usare CSS per stilizzarlo. La property font-style puo’ rendere il contenuto italic, font-weight puo’ renderlo bold, text-decoration puo’ aggiungere sottolineature, ecc.

Gli elementi <i> (italics) e <b> (bold) erano originariamente parte di HTML come elementi puramente di styling, senza significato semantico (e quindi non dovrebbero essere usati). L’HTML moderno attribuisce loro un significato semantico: <i> e’ usato per contenuto “set apart” (ma per un motivo coperto da un altro elemento), e <b> per contenuto che “merita attenzione” (ma per un motivo non coperto da altri elementi). Poiche’ il significato e’ vago, spesso e’ meglio non usarli per evitare confusione. Se vuoi mettere in italic del testo, pensa al motivo: e’ perche’ e’ enfatizzato (quindi <em>) o perche’ e’ un termine da definire o un’opera citata? C’e’ quasi sempre un elemento migliore e piu’ informativo per annotare il contenuto.

Immagini e media

Anche se le web page sono principalmente un medium testuale, spesso contengono molte informazioni puramente visive. Questo include media come immagini o video, ma anche etichette visive come icone (ad es. un bottone “search” con una lente o un hamburger button). Queste feature possono essere create con diversi elementi, ma servono attenzioni specifiche per garantire che siano accessibili e possano essere percepite dagli screen reader.

La forma piu’ comune di media visivo sul web sono le immagini, create con l’elemento <img>. Come notato in HTML Fundamentals, l’attribute src specifica il path del file immagine da mostrare. Di solito e’ un relative path verso un file servito dallo stesso domain (ad es. src="imgs/my_pic.jpg"). Ma e’ anche possibile usare un path assoluto verso un’immagine su un altro domain (ad es. src="http://pictures.com/cat_pic.jpg") — questo e’ chiamato hotlinking di un’immagine. L’hotlinking e’ spesso considerato una bad practice — non solo costringe un altro domain a fornire bandwidth per i contenuti del tuo sito, ma significa anche che il file servito potrebbe cambiare senza che tu lo sappia, causando contenuti indesiderati sul tuo sito.

Per rendere le immagini accessibili, includi sempre un attribute alt che fornisce testo alternativo quando l’immagine non puo’ essere mostrata (ad es. con screen reader, ma anche se l’immagine non si carica):

<img src="puppy_picture.jpg" alt="a cute puppy">

Questo elemento verra’ letto dagli screen reader come “a cute puppy, image”. Nota che l’“alt-text” non dovrebbe includere testo introduttivo come “a picture of”, perche’ gli screen reader indicano gia’ che si tratta di un’immagine. La WAI (Web Accessibility Initiative) fornisce ulteriori indicazioni su come scrivere un buon alt text.

Per immagini piu’ complesse (come chart o infographic), puoi anche fornire un hyperlink a una descrizione piu’ lunga usando l’attribute longdesc. Questo attribute prende un valore URI (relativo o assoluto; quello che metteresti in href) che punta a dove si trova la descrizione. Gli screen reader offriranno all’utente l’opzione di navigare verso questa descrizione lunga.

<img src="accessibility_infographic.png"
     alt="an infographic showing how to make web pages accessible"
     longdesc="infographic_text.html"> <!-- link to other page with text description -->

Ovviamente, fornire testo descrittivo per le immagini puo’ essere utile per tutti gli utenti (cosi’ capiscono cosa stanno guardando). Puoi aggiungere semanticamente una caption a un’immagine mettendola dentro un elemento <figure> e poi usando un elemento <figcaption> per la caption:

<figure>
  <img src="chart.png" alt="a chart showing some information">
  <figcaption>
    A caption for the above figure. It provides the same information,
    but in a text format.
  </figcaption>
</figure>

Gli elementi <figure> e <figcaption> sono un buon esempio di elementi semantici usati insieme: un <figcaption> puo’ essere usato solo dentro un <figure> (e deve essere il primo o l’ultimo child di quell’elemento). Se vuoi regolare aspetto o posizionamento della caption, userai CSS.

E’ anche possibile incorporare elementi video o audio in una web page usando rispettivamente <video> e <audio>. Questi elementi si aspettano una “source”, specificata con l’attribute src o con un elemento figlio <source> (usato per avere piu’ sorgenti):

<!-- esempio base dell'elemento <video>, adattato da MDN -->
<video controls width="400">
  <source src="media/videos/cat-video.mp4" type="video/mp4" />
  <!-- text to show if video isn't supported -->
  Video not supported. <a href="/media/videos/cat-video.mp4">download the video</a>
</video>

Sia <video> che <audio> supportano molti attribute aggiuntivi (come controls nell’esempio sopra) usati per personalizzare presentazione, controlli e playback. Una discussione completa di tutte le opzioni e’ fuori dallo scopo di questo testo; vedi la documentazione per dettagli ed esempi.

Nota che i controlli video in particolare potrebbero non essere completamente accessibili agli screen reader a seconda del browser. Come minimo, dovresti aggiungere caption e subtitle ai video (fornendo elementi <track>) per assicurarti che tutti gli utenti possano accedervi.

Infine, l’elemento <canvas> e’ usato per generare immagini grafiche (2d o 3d) via JavaScript. E’ la base HTML per molte interazioni grafiche o web game — anche se gran parte del lavoro avviene in JavaScript e altri linguaggi.

Headings

Gli elementi Heading (<h1>, <h2>, fino a <h6>) sono usati per dare struttura a una web page, permettendo a utenti e screen reader di navigare facilmente tra le sezioni del contenuto. Gli elementi <h#> sono block-level e di default sono piu’ grandi e bold del testo normale (con heading piu’ piccoli resi piu’ piccoli). Tuttavia, non usare gli heading solo per cambiare la formattazione del testo — usa CSS per quello.

Fai attenzione a non confondere un heading (<h1>) con l’elemento <head> (per i meta-data) o con l’elemento di sectioning <header> descritto sotto.

Gli heading sono usati da browser e screen reader per costruire automaticamente un “table of contents” della pagina. Quindi dovresti considerarli come titoli di capitoli o sezioni — come etichetteresti le parti di un “outline” della pagina. Ogni web page dovrebbe avere un solo <h1>, che rappresenta il “titolo” della pagina. Sezioni diverse possono essere marcate con <h2>, mentre le sottosezioni con <h3> e cosi’ via. Una pagina puo’ avere molti <h2> o <h3>, ognuno che segna un “livello” nell’outline. Per esempio, le sezioni di questo capitolo sono annotate con heading <h#>:

<h1>Semantic HTML</h1>
<h2>Specific HTML Elements</h2>
<h3>Hyperlinks</h3>
<h3>Inline Textual Elements</h3>
<h3>Images and Media</h3>
...
<h2>ARIA for Accessibility</h2>

E’ possibile che una pagina non abbia sezioni o sottosezioni, quindi potrebbe non avere <h2> o elementi piu’ piccoli.

Poiche’ screen reader e altri sistemi usano i heading per generare un “table of contents” (che aiuta a navigare velocemente la pagina), e’ importante che i heading siano sia significativi che gerarchici. Devono essere significativi nel descrivere il contenuto della sezione che introducono, come un titolo di sezione. Devono essere gerarchici nel non saltare livelli: ogni <h3> deve avere un <h2> sopra — non puoi avere una sottosezione senza una sezione.

Vedi anche Sectioning Elements per maggiori dettagli sulla organizzazione strutturale del contenuto.

Elementi non semantici

Anche se lo scopo di HTML e’ fornire informazioni semantiche sul contenuto, ci sono elementi che non hanno un significato o aspetto proprio. L’elemento <div> (division) serve a “raggruppare” altri contenuti, fornendo un parent o container condiviso per quel contenuto. E’ un block element che non ha impatto semantico o di aspetto da solo; serve solo a organizzare contenuto nell’albero degli elementi HTML.

<!-- these three paragraphs will all have the same
     "parent" which can be styled on its own -->
<div>
  <p>Lorem ipsum dolor</p>
  <p>sit amet consectetur</p>
  <p>Adipisicing elit</p>
</div>

<!-- these paragraphs are in a different "group" -->
<div>
  <p>Lorem ipsum dolor</p>
  <p>sit amet consectetur</p>
</div>

Questo e’ particolarmente utile per organizzare contenuti e applicare styling con CSS:

<!-- A collection of "cards" on a page -->
<div class="card-group">
  <div class="card">
    <div class="card-header">Featured</div>
    <div class="card-body">
      <h2>Card 1</h2>
      <p>An information card</p>
    </div>
  </div>

  <div class="card">
    <div class="card-body">
      <h2>Card 2</h2>
      <p>Another information card</p>
    </div>
  </div>
</div>

Nell’esempio sopra, gli elementi <div> ricevono class CSS diverse per essere stilizzati in modo differente: gli elementi con class card possono essere stilizzati per sembrare una “card” di contenuto, mentre quelli con class card-header possono sembrare header di quelle card. Come nell’esempio, e’ molto comune avere <div> annidati (un <div> dentro un <div> dentro un <div>). In effetti, <div> e’ forse l’elemento HTML piu’ comune sul web.

Ricorda che un <div> non fornisce alcuna informazione semantica; eventuali cambiamenti di stile applicati tramite esso saranno percepibili solo dagli utenti vedenti. Per questo e’ sempre bene chiedersi se un altro block element — come un paragraph, un heading o un sectioning element — possa essere piu’ appropriato (e potrebbe non esserlo).

Allo stesso modo, un elemento <span> e’ come un <div> (non ha significato semantico o impatto visivo proprio), ma e’ un elemento inline, quindi si usa all’interno del testo per marcare brevi passaggi:

<p>Baa baa <span class="dark">black</span> sheep,
   have you any wool?</p>

Un <span> puo’ essere usato per applicare styling a contenuto inline, ad esempio per rendere una certa parola di colore diverso. Ma poiche’ di solito c’e’ un motivo per questi cambiamenti di stile, e’ quasi sempre meglio usare un elemento testuale semantico appropriato. Per esempio, se la parola e’ importante, usa <em>. Puoi pensare a <span> come all’elemento da usare quando serve styling inline ma nessun altro elemento e’ appropriato.

Sectioning elements

Gli elementi heading <h#> fanno parte della specifica HTML originale, quindi sono supportati da tutti i sistemi di screen reader. Tuttavia, HTML 5 ha introdotto elementi aggiuntivi per organizzare il contenuto della pagina e rendere la struttura piu’ esplicita. Questi sono spesso chiamati sectioning elements. Sono tutti elementi block-level che non hanno impatto sull’aspetto, ma forniscono informazioni semantiche sull’organizzazione del loro contenuto. Puoi vederli come <div> specializzati — raggruppano elementi, ma danno anche un significato al motivo del raggruppamento.

Sono spesso chiamati semantic elements o “sectioning elements”. Sono tutti elementi block-level che non producono effetti visivi da soli, ma forniscono struttura semantica al contenuto web. Puoi pensarli come <div> specializzati.

Gli elementi di sectioning comuni sono descritti qui sotto.

  • Un <section> rappresenta una sezione specifica di contenuto. In particolare, una “section” e’ una porzione di contenuto che sarebbe identificata da un heading (ad es. un <h2> o <h3>) — una section dovrebbe quasi sempre contenere un heading come child. Quindi questo elemento puo’ essere usato per raggruppare semanticamente tutto il contenuto di quella sezione (che sta sotto l’heading). Se un gruppo di contenuti non avrebbe un heading, probabilmente non e’ un <section> (e sarebbe piu’ probabilmente un <div>).

    <section>
  <h2>HTML</h2>
  <p>...</p>
  <section>
    <h3>Introduction</h3>
    <p>....</p>
  </section>
  <section>
    <h3>Syntax</h3>
    <p>....</p>
  </section>
</section>

    Nota che, come nell’esempio sopra, gli elementi <section> possono contenere altri <section> (“subsection”), ciascuno con un heading appropriato.

    Ricorda che <section> non ha aspetto di default, anche se puoi usare CSS per personalizzarlo. Sono usati per fornire semantica organizzativa utilizzabile da screen reader e altri sistemi.

    <section> e’ il piu’ generico dei sectioning elements; gli altri possono essere visti come tipi di section piu’ specifici.

  • Un <header> rappresenta contenuto introduttivo, come heading, titoli o banner image. Pu puo’ includere anche elementi comuni della pagina come navigazione o search bar.

    Fai attenzione a non confondere un <header> con un heading (<h1>) o con l’elemento <head>. Un <header> e’ un sectioning element per contenuto introduttivo mostrato nella pagina e puo’ contenere piu’ child element. Il <head> e’ un elemento che NON fa parte del <body> (quindi non e’ mostrato nella pagina) e contiene metadata sulla pagina.

    Una pagina dovrebbe quasi sempre contenere un <header> con contenuto introduttivo, come l’<h1> e banner image. Questo uso implica il landmark role role="banner" (vedi ARIA for Accessibility sotto). Inoltre, specifici <section> possono includere un <header> per contenere il contenuto introduttivo di quella sezione.

    <body>
  <!-- this is is the header ("banner") for the entire page -->
  <header>
    <h1>Page Title</h1>
  </header>
  <section>
    <!-- this is the header for this section -->
    <header>
      <h2>About Section<h2>
    </header>
  </section>
</body>

    Anche se e’ possibile fornire un <header> per ogni sezione (e ogni <section> dovrebbe avere un heading), in pratica una <section> con poco contenuto introduttivo — come un solo heading — spesso non avra’ un <header> separato perche’ non aggiunge molta informazione sulla struttura della pagina. Puo’ essere utile ma non e’ obbligatorio.

  • Un <footer> di solito contiene informazioni sull’autore o sulla fonte dei contenuti della pagina, includendo informazioni di copyright. Questo elemento implica il landmark role role="contentinfo" (vedi ARIA for Accessibility sotto).

    Un <footer> contiene comunque contenuto visibile, quindi va dentro il <body>! Tipicamente le pagine hanno un solo <footer> — e’ posizionato in fondo alla pagina come ultimo elemento del <body>, ma non sempre (usare questo sectioning element aiuta a navigare la pagina anche se il <footer> e’ altrove). Il <footer> contiene informazioni visibili sulla pagina, come autore e copyright. Non contiene la “conclusione” della pagina (quella sarebbe una normale <section>). Le informazioni dell’autore sono spesso dentro un elemento <address>, che rappresenta informazioni di contatto.

    <footer>
  Created by <address><a href="mailto:joel@uw.edu">joel@uw.edu</a></address>.
  &copy; 2024
</footer>

    Anche i singoli <section> possono avere il loro <footer>, ma non e’ comune perche’ le sezioni raramente hanno informazioni d’autore diverse dal resto della pagina. E’ piu’ comune che un elemento <article> abbia un proprio <footer>.

  • L’elemento <main> rappresenta il contenuto “principale” o centrale del documento. Una web page puo’ avere un solo <main>. Questo elemento implica il landmark role role="main" (vedi ARIA for Accessibility sotto).

    La maggior parte delle pagine ha un <main>. Di solito viene dopo il <header> (ma non dentro — infatti, <main> non puo’ essere discendente di <header>). L’elemento <main> contiene poi i <section> della pagina, ed e’ seguito dal <footer>:

    <body>
  <header>
    introductory content
  </header>

  <main>
    ...main content
    <section>
      ...section content
    </section>
    ... more content (including more sections)
  </main>

  <footer>
    author and copyright information
  </footer>
</body>

    Questa e’ una struttura tipica per un documento HTML.

  • Un <nav> rappresenta la parte di una pagina che fornisce link di navigazione, come una navigation bar o un table of contents. Non tutti i link devono stare in un <nav>; questo elemento rappresenta una sezione della pagina puramente navigazionale (pensa a un <section> specializzato). Questo elemento implica il landmark role role="navigation" (vedi ARIA for Accessibility sotto). Un <nav> si trova spesso nel <header> (per la nav bar in cima), ma non e’ obbligatorio — una pagina puo’ avere piu’ sezioni <nav>.

  • Un <article> e’ simile a un <section>, ma rappresenta contenuto autonomo che potrebbe essere pubblicato indipendentemente — come un news article o un blog post. Se una sezione probabilmente verrebbe pubblicata come documento a se’, allora puo’ essere un <article>. Nota che un <section> puo’ raggruppare piu’ <article> (come un blog roll), e un <article> puo’ contenere piu’ <section>. Questo rispecchia come e’ strutturato un giornale (ad es. con una “Sports Section” che contiene articoli, che a loro volta possono avere sezioni).

In generale, usare sectioning elements semantici aiuta a strutturare chiaramente il contenuto, oltre a organizzarlo per gli screen reader e renderlo piu’ accessibile. Quasi ogni pagina dovrebbe avere <header>, <main> e <footer>; pagine con piu’ contenuto saranno divise in <section>; e siti con piu’ pagine organizzeranno la navigazione con un <nav>.

Tabelle

L’elemento <table> rappresenta dati tabellari, come una data table tipo foglio di calcolo. Le tabelle sono una delle strutture piu’ complesse in HTML e richiedono diversi elementi e attribute. Questa sezione fornisce un breve esempio per creare tabelle semplici e accessibili; per piu’ esempi e guidance vedi la documentazione MDN per <table> e la MDN Learn HTML tables guide.

In HTML, una tabella e’ composta da righe, ognuna delle quali contiene un numero di celle. Celle corrispondenti tra righe implicano una colonna, anche se le colonne non sono indicate esplicitamente in HTML.

A diagram showing the rows, columns, and cells of a table.

Le tabelle sono codificate in HTML con l’elemento <table>. Un <table> contiene piu’ elementi <tr> (table row) che rappresentano le righe. Ogni riga <tr> contiene piu’ elementi <td> (table datum) che rappresentano le singole celle.

<table> <!-- whole table -->
  <tr> <!-- a row -->
    <td>Ada</td> <!-- a cell -->
    <td>64</td>
    <td>135</td>
  </tr>
  <tr>
    <td>Bob</td>
    <td>74</td>
    <td>156</td>
  </tr>
</table>
La tabella di dati renderizzata prodotta dal codice HTML.

Un elemento <th> (table header) rappresenta una cella che si trova nella riga o colonna di intestazione, e puo’ sostituire un <td> (e’ comunque una cella, ma specializzata). Per esempio, la prima riga di una tabella puo’ contenere <th> invece di <td> perche’ quelle celle sono le intestazioni delle colonne. Di default, <th> renderizza il contenuto in bold. Un <th> si aspetta un attribute scope che indica se e’ l’header di una "row" o di una "col" (colonna). Anche se scope puo’ essere ridondante per tabelle semplici, e’ una best practice includerlo per garantire che tecnologie assistive come gli screen reader presentino e navighino la tabella correttamente.

<table>
  <tr>
    <th scope="col">name</th> <!-- header cell -->
    <th scope="col">height</th>
    <th scope="col">weight</th>
  </tr>
  <tr>
    <td>Ada</td>
    <td>64</td>
    <td>135</td>
  </tr>
</table>

Le tabelle possono essere rese piu’ accessibili includendo elementi di table sectioning (simili concettualmente a <header>, <main> e <footer>). L’elemento <thead> raggruppa le righe di intestazione della tabella con informazioni sulle colonne (di solito con celle <th>); l’elemento <tbody> raggruppa le righe del corpo principale della tabella; l’elemento <tfoot> raggruppa le righe in fondo alla tabella con informazioni sulle colonne (ad es. dati di riepilogo). Questi elementi non impattano l’aspetto, ma aiutano a organizzare e presentare la tabella. Sono anche la cosa piu’ simile a usare un <div> per stilizzare sezioni diverse della tabella.

<table>
  <thead> <!-- the "header" section -->
    <tr>
      <th>name</th>
      <th>height</th>
      <th>weight</th>
    </tr>
  </thead>
  <tbody> <!-- the "main" section -->
    <tr>
      <td>Ada</td>
      <td>64</td>
      <td>135</td>
    </tr>
  </tbody>
</table>

Le celle (<td> o <tr>) possono estendersi su piu’ righe o colonne specificando gli attribute colspan e rowspan. Se una cella si estende su piu’ colonne o righe, allora ometti le celle <td> che sta sostituendo. Nota che lo spanning di righe o colonne non e’ comune nei dati tabellari (poiche’ un dato non dovrebbe estendersi su piu’ feature o osservazioni) e puo’ rendere difficile per chi usa screen reader capire la struttura dei dati. E’ meglio organizzare i dati per non richiedere questo spanning.

Come per altri elementi, le tabelle si stilizzano principalmente con CSS; gli attribute HTML dovrebbero essere usati solo per la struttura della tabella. Nota che le regole CSS di solito si applicano alle celle (<td>, <th>) e non alle righe o sezioni — molte property inaspettatamente non sono ereditate dalle righe, quindi vanno applicate direttamente alle celle. Ci sono anche alcune property CSS specifiche per tabelle, come border-collapse e border-spacing (usate per lo spacing tra celle).

Non usare mai un <table> per strutturare documenti e posizionare contenuti. <table> va usato solo per contenuti che sono semanticamente una tabella (ad es. data table). Se vuoi disporre contenuti in colonne o griglia, usa property CSS come flexbox o grid. Ricorda che HTML serve per la semantica, non per l’aspetto.

Form

Nelle web page, un form e’ una parte della pagina che permette all’utente di interagire fornendo informazioni. Gli utenti possono inserire input (testo scritto, caselle selezionate, ecc.) che vengono poi submitted al web server. I form sono una parte comune delle web page che usi ogni giorno — e potresti aver interagito con essi senza accorgertene. Come le tabelle, i form sono strutture complesse in HTML; questa sezione descrive come creare form di base e come renderli accessibili. Per piu’ esempi e guidance vedi la guida MDN ai web form.

I web form si definiscono con l’elemento <form>. Questo block element annota una parte della pagina che accetta dati inseriti dall’utente (tramite <input>, descritti sotto) e specifica cosa fare con quei dati quando il form viene submit.

Quando un form viene submit (ad es. cliccando un <button> di tipo submit), il browser raccoglie i dati inseriti e li invia al server tramite una HTTP request — simile alla request inviata quando l’utente clicca un hyperlink (questo significa che inviare un form carica una nuova pagina). Dove e come i dati vengono inviati dipende da due attribute dell’elemento <form>: action e method.

<!-- a very simple form -->
<form action="result.html" method="GET">
  <!-- labeling text for the input -->
  <label for="greetInput">Provide a greeting:</label>

  <!-- Where the user provides input data -->
  <input type="text" id="greetInput" name="greeting">

  <!-- Submit button -->
  <button type="submit">Submit</button>
</form>

L’attribute action specifica l’URL a cui i dati saranno inviati quando il form viene submit. Di solito e’ un URL nello stesso domain della pagina corrente, quindi si fornisce solo un relative path. Quindi nell’esempio sopra, i dati saranno inviati all’URL result.html; il server gestira’ i dati inviati e poi restituira’ una response per quell’URL (in pratica, caricando quella pagina). Nota che i server possono fare molte cose con i dati, incluso redirigere gli utenti, quindi la pagina che si carica per l’utente non e’ necessariamente la stessa specificata in action.

L’attribute method indica quale HTTP Method usera’ la request risultante. Opzioni valide sono "GET" (per request GET — usate quando si recuperano dati dal server) o "POST" (per request POST — usate quando si aggiungono o aggiornano dati sul server). I form che inviano GET trasmettono i dati come query parameter (uno per ogni <input>) aggiunti all’URL action — ad esempio, inviare il form sopra con il messaggio “Ciao” invierebbe una HTTP request a result.html?greeting=Ciao. I form che inviano POST includono i dati nel body della HTTP request (da cui il server puo’ leggerli).

Le app React e sistemi simili spesso gestiscono i dati del form client-side; quindi gli attribute action e method vengono omessi perche’ non sono usati. Puoi impedire a un form di inviare una nuova HTTP request usando event.preventDefault() in JavaScript; vedi DOM Manipulation per dettagli.

Elementi <input>

Gli utenti forniscono dati ai form tramite elementi <input>. Un <input> fornisce un controllo interattivo, come un campo di testo o una checkbox. Gli elementi <input> sono inline e empty — come un <img>, non hanno contenuto testuale proprio.

<!-- elementi di input di esempio -->
<label for="usernameInput">Username:</label>
<input type="text" id="usernameInput" name="username">

<label for="passwordInput">Password:</label>
<input type="password" id="passwordInput" name="password">

Ci sono molti tipi di <input>, specificati dall’attribute type, inclusi:

  • text: un campo di testo monoriga in cui l’utente puo’ digitare. E’ il type di default per un <input>.
  • password: come text, ma il testo digitato e’ offuscato (ad es. punti o asterischi).
  • checkbox: una casella che l’utente puo’ selezionare o deselezionare. Nota che non c’e’ testo associato all’input; devi usare un <label> per dire cosa sta selezionando.
  • radio: un pulsante selezionabile da un set (il “cerchio”). Si chiama “radio button” perche’ come una radio puoi selezionare solo un canale. I radio button sono raggruppati dando loro lo stesso attribute name.
  • number: un campo in cui l’utente puo’ inserire un numero
  • date: un input in cui l’utente puo’ selezionare una data. La maggior parte dei browser fornisce un calendario a discesa, ma permette anche di digitare la data. Un input time funziona in modo simile per l’ora del giorno, e un input datetime-local permette di selezionare entrambi.
  • file: un input in cui l’utente puo’ scegliere un file dal sistema operativo. Di default appare come un button che, se cliccato, apre un’interfaccia di scelta file.
  • hidden: un input non mostrato all’utente, ma i cui dati vengono comunque inviati con il form. Gli input hidden si usano per dati non specificati o modificati direttamente dall’utente, ma generati dinamicamente dalla pagina o dal sistema (ad es. via JavaScript).

Ogni <input> puo’ supportare attribute aggiuntivi in base al tipo. Per esempio, un input number supporta attribute min e max che limitano i numeri consentiti. Questi attribute possono essere usati anche per la form validation client-side, informando gli utenti se i valori inseriti sono invalidi.

Anche se <input> e’ l’elemento piu’ comune per raccogliere input in un <form>, ce ne sono altri. Un elemento <textarea> fornisce una casella di testo piu’ grande in cui l’utente puo’ scrivere piu’ righe. Un elemento <select> fornisce un menu a discesa; ogni voce e’ specificata con un elemento figlio <option>:

<!-- A simple select dropdown -->
<select name="animal">
  <option value="">Choose an Animal</option>
  <option value="lion">Lion</option>
  <option value="tiger">Tiger</option>
  <option value="bear">Bear</option>
</select>

Vedi la documentazione per altre opzioni nell’uso di questi elementi. Collettivamente, <input>, <select> e simili sono chiamati form control elements.

Come negli esempi sopra, tutti i form control e gli input supportano e si aspettano un attribute name. Questo attribute fornisce un identificatore per distinguere l’input dagli altri nel form (il form stesso non sa quale testo descrittivo sia attorno o etichetti gli input). Nota che name e’ distinto dall’attribute id — name identifica il valore inserito dall’utente nei dati del form, mentre id identifica l’elemento <input> stesso (indipendentemente dai dati inseriti). Poiche’ gli id sono usati con <label> per accessibilita’, molti <input> avranno entrambi gli attribute — possono anche usare lo stesso identificatore, sembrando ridondanti, ma servono a scopi diversi.

Tutti gli input supportano anche un attribute value. Questo attribute memorizza il valore inserito o selezionato dall’utente. Puoi usarlo per fornire un valore di default che l’utente puo’ sostituire.

<!-- input di esempio con nomi diversi e valori di default -->
<label for="firstnameInput">First Name:</label>
<input type="text" id="firstnameInput" name="firstname" value="Joel">

<label for="passwordInput">Last Name:</label>
<input type="text" id="lastnameInput" name="lastname" value="Ross">

Gli input di testo supportano anche un attribute placeholder, che fornisce testo iniziale mostrato all’utente ma non e’ un valore di default (il valore resta vuoto "").

Checkbox, radio button e select dropdown usano l’attribute value per distinguere tra opzioni selezionate:

<!-- a set of radio buttons -->
<label for="smallRadio">Small</label>
<input type="radio" id="smallRadio" name="size" value="small">
<label for="mediumRadio">Medium</label>
<input type="radio" id="mediumRadio" name="size" value="medium">
<label for="largeRadio">Large</label>
<input type="radio" id="largeRadio" name="size" value="large">

Quando il form viene submit, i dati inseriti dall’utente sono codificati usando l’attribute name come identificatore o “chiave”, e l’attribute value come valore. Questo si vede nei query parameter dell’URL target di una request GET. Per esempio, senza modifiche dell’utente, gli input firstname e lastname dell’esempio sopra produrrebbero una lista di query parameter tipo: ?firstname=Joel;lastname=Ross — le chiavi (prima di =) sono i name, e i valori (dopo =) sono i value. Di conseguenza i name sono spesso scelti per produrre identificatori di dati facili da consumare per un server.

Oltre a <input> e affini, un <form> contiene spesso elementi <button>. In particolare un <button type="submit"> inviera’ il form parent quando cliccato. E’ una best practice avere un submit button esplicito (anche per form di ricerca) e non affidarsi solo all’utente che preme “enter” — questo rende il form piu’ accessibile.

Label e accessibilita’

Essendo strutture complesse e chiave per l’interazione, i form richiedono particolare cura per essere accessibili.

Uno dei passaggi piu’ importanti per rendere un form accessibile e’ fornire label per tutti i form control e input. Le label si specificano usando l’elemento <label> (che vedi negli esempi sopra). Il testo dentro <label> e’ quello che l’utente vede e che descrive l’input (spesso istruzioni su quali dati inserire). Associ un <label> a un input specifico usando l’attribute for del <label> — questo attribute prende come valore l’id dell’input (l’id identifica univocamente un elemento nella pagina):

<!-- this label is a description for the element with the `nameInput` id -->
<label for="nameInput">Name:</label>
<input type="text" id="nameInput" name="name">

Di default, quando l’utente clicca il testo del <label>, il form control associato riceve focus (viene selezionato, con il cursore pronto a ricevere input). Questo rende piu’ facile selezionare l’input giusto, ed e’ particolarmente utile per elementi piccoli come le checkbox, che possono essere difficili da colpire (per via della Fitts’ Law).

Tutti i form control dovrebbero essere etichettati per supportare screen reader e altre tecnologie assistive. Se non vuoi testo descrittivo per utenti vedenti (per altri indizi visivi), puoi usare CSS per nascondere visivamente il <label>, ma dovrebbe essere sempre presente.

E’ anche possibile associare implicitamente un <label> e un <input> rendendo l’<input> un child element del <label>. Questo puo’ essere utile per contenuti generati dinamicamente quando non puoi specificare un id per l’<input> (ad es. form in cui l’utente aggiunge text box). Tuttavia, l’associazione esplicita con l’attribute for e’ meglio supportata dalle tecnologie assistive, quindi va usata quando possibile.

<!-- implicitly labeling an input, used only when `id` cannot be specified -->
<label>
  Name:
  <input type="text" name="name">
</label>

Mantenere i form brevi e semplici (con meno input possibile) e’ buono sia per l’accessibilita’ sia per l’usabilita’ complessiva — in genere gli utenti preferiscono form brevi. Per form piu’ complessi, puoi anche raggruppare input mettendoli insieme dentro un elemento <fieldset>. Questo e’ particolarmente utile per i radio button. L’elemento <legend> e’ usato per dare un “titolo” a un <fieldset>.

<fieldset>
<legend>Programming Language</legend>
  <div>
    <input type="radio" name="language" id="htmlInput" value="html">
    <label for="htmlInput">HTML</label>
  </div>
  <div>
    <input type="radio" name="language" id="cssInput" value="css">
    <label for="cssInput">CSS</label>
  </div>
</fieldset>

Per maggiori indicazioni su form accessibili, vedi il WAI Forms Tutorial.

5.2 ARIA for Accessibility

Come notato in Client-Side-Development, rendere accessibili le web page implica conformarsi alle Web Content Accessibility Guidelines (WCAG) del W3C, sviluppate dalla Web Accessibility Initiative (WAI) (parte del W3C). Il WAI ha anche creato uno standard web aggiuntivo per supportare l’accessibilita’. Questo standard si chiama Accessible Rich Internet Applications Suite, o ARIA. ARIA specifica un estensione a HTML, definendo attribute HTML aggiuntivi che possono essere inclusi negli elementi per fornire supporto extra agli screen reader e ad altre tecnologie assistive. Questi attribute aiutano gli screen reader a navigare una pagina (per chi non puo’ semplicemente scrollare visivamente), e forniscono un meccanismo con cui web app “ricche” e interattive (come quelle che creerai con JavaScript) possono comunicare il loro comportamento agli screen reader.

Questa sezione descrive alcuni degli attribute ARIA piu’ comuni che puoi usare per assicurare che le web page siano accessibili a tutti. Noterai che molti attribute ARIA sono resi ridondanti dall’uso di HTML semantico — HTML come standard e’ evoluto per supportare meglio l’accessibilita’ da quando ARIA e’ stato creato. Detto questo, ci sono ancora casi in cui HTML non basta, e ARIA deve intervenire.

ARIA Labeling

Un passo fondamentale per rendere le web page accessibili e’ assicurarsi che siano perceivable per gli screen reader. Questo significa che il contenuto non puo’ essere solo visivo. L’esempio piu’ comune e’ includere attribute alt per gli elementi <img>, che permette a elementi visivi di essere percepiti e compresi dalle tecnologie assistive.

Ma l’attribute alt e’ supportato solo da <img>, anche se altri elementi possono essere usati per informazioni visive (ad es. un <div> con un background image, o un <svg> con grafica). ARIA permette di fornire “alternate text” per altri elementi usando l’attribute aria-label, che specifica quale testo deve essere letto al posto del contenuto normale.

<div class="green-rect" aria-label="a giant green rectangle"></div>

Nota che aria-label serve solo per elementi puramente visivi (senza testo); non devi mettere aria-label su ogni <div> o su qualsiasi <p>. Se l’elemento contiene testo — anche se annidato tra piu’ child — probabilmente non serve un aria-label.

Per descrizioni piu’ lunghe (piu’ frasi), puoi usare aria-describedby con un riferimento all’id di un altro elemento nella stessa pagina che contiene la descrizione testuale. aria-describedby prende come valore un riferimento a fragment (simile a un bookmark hyperlink) verso l’elemento contenente la descrizione.

<div class="green-rect" aria-describedby="#rectDetail"></div>
<p id="rectDetail">The above rectangle is giant and green.</p>

Il testo descrittivo puo’ stare ovunque nella pagina; e’ ok che un elemento visivo sia separato dalla descrizione da altri elementi. Come per i bookmark link, nota che il valore aria-describedby inizia con il simbolo #, ma l’id dell’elemento target non lo include. Inoltre, nota che aria-label sostituisce il contenuto dell’elemento, mentre aria-describedby viene letto dopo il contenuto dell’elemento, perche’ fornisce piu’ informazioni (confronta con l’attribute simile aria-labelledby).

In generale, il modo migliore per rendere accessibili le informazioni puramente visive e’ non avere informazioni puramente visive in primo luogo. Il contenuto di una pagina dovrebbe sempre supportare utenti e browser non visivi: includi descrizioni testuali e caption per le immagini, usa etichette testuali invece di icone per i button, ecc. Questo e’ il modo migliore per supportare il universal design.

Mentre alt e aria-label permettono di rendere percepibili elementi visivi, ci sono immagini (o altri elementi) puramente decorativi: loghi, icone che accompagnano testo nei button, ecc. Puoi rendere questo contenuto piu’ accessibile e fare in modo che uno screen reader lo “salti” usando un altro attribute ARIA aria-hidden="true". Un elemento con questo attribute appare comunque nella pagina renderizzata, ma viene ignorato dagli screen reader. (Questo attribute e’ un’eccezione alla regola generale che gli attribute boolean non prendono valori).

<!-- a search button with an icon -->
<!-- the icon img will not be read, but the button text will be -->
<button>
  <img src="search_icon.png" aria-hidden="true">
  Search
</button>

Nota che elementi nascosti via CSS (ad es. con display:none) sono gia’ rimossi da cio’ che uno screen reader legge, quindi non richiedono l’attribute ARIA aggiuntivo.

ARIA Roles and Landmarks

La specifica ARIA definisce un attribute role che dettaglia lo scopo semantico di un elemento per gli screen reader quando quello scopo e’ chiaro solo tramite styling visivo.

Per esempio, una pagina puo’ avere un <div> o un <svg> stilizzato per sembrare e comportarsi come un button; l’attribute ARIA aiuta gli screen reader a capire che quell’elemento puo’ essere premuto per causare un’azione:

<div class="btn" role="button">Submit</div>

Dal punto di vista dello screen reader, questo elemento agisce come un <button>, anche se e’ un elemento diverso per qualche motivo.

Ci sono molti ARIA role diversi, inclusi role per struttura e per widget interattivi come dialog, switch o menu. Alcuni, come button nell’esempio sopra, sono ridondanti con elementi semantici introdotti dopo lo standard ARIA. E’ meglio usare elementi che riflettono il loro scopo semantico invece di affidarsi a ARIA role. Se un elemento deve essere cliccato per causare un’azione, usa un <button> invece di role="button" — poi stilizzalo con CSS come vuoi.

Inoltre, l’attribute ARIA role puo’ essere usato per indicare landmark regions. Un landmark role indica posizioni specifiche nel documento e consente a uno screen reader di “saltare” facilmente a quella sezione. Questo aiuta a garantire che la pagina sia navigable. I landmark role includono:

  • banner: contenuto relativo al sito nel suo insieme, piuttosto che alla pagina specifica (nome derivato dal “banner” in cima alla pagina). L’elemento <header> implica questo role.
  • main: il contenuto “main”. Utile per far saltare lo screen reader al cuore della pagina, oltre banner e navigazione. L’elemento <main> implica questo role.
  • contentinfo: sezione con informazioni sulla pagina, come contatti autore e copyright. Di solito sta in fondo. L’elemento <footer> implica questo role.
  • navigation: sezione di link per muoversi nella pagina, come una nav bar. Permette agli screen reader di saltare ai “quick links” e muoversi rapidamente. Nota pero’ che ascoltare tutti i link puo’ richiedere tempo, quindi e’ importante permettere di saltare al resto del contenuto (con una tecnica chiamata “skip link”). L’elemento <nav> implica questo role.

Come detto, questi landmark role sono anche impliciti nei sectioning elements; non serve includere anche un role su questi elementi, ed e’ best practice usare gli elementi espliciti invece dei landmark role!

Considerare il contenuto in termini di questi role puo’ essere un buon trucco di design per organizzare la pagina in modo comprensibile e accessibile per tutti!

ARIA e interattivita’

ARIA fornisce supporto all’accessibilita’ anche per web page interattive, dove il contenuto cambia mentre la pagina viene usata, attraverso attribute che notano esplicitamente lo stato della web application.

Per esempio, considera una pagina con un hamburger menu) o un dropdown/“slide-out” simile. Le tecnologie assistive devono informare l’utente se il menu e’ espanso o se il contenuto scorrevole e’ visibile. Questo puo’ essere gestito con ARIA state attributes come aria-expanded:

<!-- un esempio di menu hamburger con attributi ARIA -->
<button aria-expanded="false" aria-controls="menuContent" aria-label="Expand Menu">
  <img src="hamburger-icon" alt="a hamburger menu">
</button>

<ul id="menuContent" class="collapsed">
  <li>Link 1</li>
  <li>Link 2</li>
  ...
</ul>

Nell’esempio sopra, l’elemento <button> controlla se il <ul> e’ espanso o no — nell’HTML corrente, il <ul> e’ collassato (nascosto) tramite una class CSS. Usando JavaScript (vedi DOM Manipulation), cliccare il button puo’ cambiare lo styling CSS del <ul> per renderlo visibile.

L’attribute ARIA aria-expanded e’ sull’elemento button per indicare alle tecnologie assistive che questo button (a) e’ un controllo interattivo per una parte della pagina, e (b) il suo stato corrente e’ “non espanso”. Quando il button viene premuto, il JavaScript cambierebbe anche l’attribute aria-expanded a "true", indicando ora agli screen reader che il menu e’ aperto. L’attribute aria-controls fa riferimento all’id dell’elemento che il button sta controllando (cosi’ lo screen reader sa quale elemento e’ stato modificato).

Widget e pattern interattivi diversi possono richiedere altri attribute ARIA per supportare al meglio le tecnologie assistive. Quando guardi esempi di questi widget (ad es. quelli di Bootstrap), verifica gli attribute ARIA appropriati e consulta la documentazione ARIA per best practice.

Il WAI afferma chiaramente che No ARIA is better than Bad ARIA. Usare male ARIA puo’ disinformare gravemente gli utenti di tecnologie assistive, rendendo piu’ difficile capire la pagina — invece di semantica poco chiara, avresti semantica sbagliata. Un buon uso di HTML semantico aiuta molto l’accessibilita’; ARIA puo’ e dovrebbe essere usato solo per colmare gap espliciti dove l’HTML semantico non basta.
