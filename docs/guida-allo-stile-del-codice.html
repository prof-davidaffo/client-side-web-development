Accessibilità dei siti web

Chapter 21 Guida allo stile del codice

Il codice e’ letto da due audience diverse: i computer, che eseguono gli statements (o renderizzano la web page), e gli esseri umani, che devono interpretare e aggiornare il codice nel tempo. Usare un buon coding style—il modo e il formato in cui il codice e’ scritto—e’ fondamentale per supportare gli esseri umani che lo leggono. Il codice scritto male puo’ essere interpretabile dal computer e quindi funzionare correttamente, ma deve anche essere comprensibile alle persone.

Un codice scritto bene (con “good style”) soddisfa due proprieta’:

  • E’ facile da leggere per le persone
  • E’ facile da modificare in futuro per le persone

Tutte le linee guida sul coding style dovrebbero seguire questi due obiettivi. Se ti chiedi se un approccio di styling e’ migliore di un altro, valutalo in termini di questi obiettivi.

Ognuno ha opinioni diverse su cosa sia “good style”; esistono infatti numerose “style guide” per i linguaggi di web programming: per esempio, Google ha le proprie style guide HTML/CSS e JavaScript, cosi’ come Mozilla. Esistono anche tools che fanno rispettare le linee guida di stile. I “linters” come ESLint identificano errori di sintassi, ma possono anche avere forti opinioni sullo stile, marcando il “poor style” come errore.

Consiglio di essere cauti con i tool o plugin automatici di code styling (“beautifier”). Anche se possono essere utili, e’ possibile che “puliscano” il codice in un modo che non lo migliora davvero. Fai molta attenzione a qualsiasi cosa che scriva o modifichi codice al posto tuo, per essere sicuro che faccia la cosa giusta!

Questo capitolo fornisce diverse linee guida specifiche seguendo le opinioni degli autori (e docenti del corso INFO 340: Client Side Development) per aiutarti a scrivere codice che sia generalmente piu’ facile da leggere e modificare—cioe’ con good style. Non vuole essere esaustivo; e’ piuttosto una raccolta di linee guida che gli studenti spesso mettono in discussione o faticano a seguire intuitivamente. Vedi le style guide linkate sopra per suggerimenti piu’ completi.

Questo capitolo e’ un work in progress, con nuove linee guida aggiunte quando emergono.

21.1 HTML Guidelines

  • Usa sempre lowercase letters per i tag HTML. Questo aiuta la leggibilita’ e la consistenza.

    <!-- Do this -->
<p>lorem ipsum</p>

<!-- Don't do this -->
<P>lorem ipsum</P>
<P>lorem ipsum</p>

Spacing

  • In generale, metti i tag dei block elements su righe separate, con il contenuto dei block elements su una riga separata (indentata)—a meno che il contenuto di quel block element sia molto breve. I block elements successivi sono indentati di un ulteriore livello. Questo rende il codice piu’ facile da leggere grazie alla struttura a blocchi e piu’ facile da modificare aggiungendo contenuto dentro i blocchi.

    <!-- Do this -->
<div>
  lorem ipsum
</div>

<!-- Do this -->
<div>
  <p>
   lorem ipsum
  </p>
</div>

<!-- Do this -->
<p>Ciao mondo</p> <!-- contenuto breve, quindi puo' stare sulla stessa riga dei tag -->

<!-- Don't do this -->
<div><p>lorem ipsum</p></div>

  • Per molti inline elements—soprattutto quelli di formattazione testo (ad es. <em>, <strong>, <a>)—e’ meglio mantenerli inline con il resto del contenuto. Pensa a come scriveresti un paragrafo, con alcune parole formattate nel mezzo. Non separare gli inline elements. Questo rende piu’ facile vederli come “inline” e permette di integrarli nel codice senza preoccuparti della spaziatura. (Questo spesso risolve anche problemi di trailing space). Nota che questa linea guida e’ una di quelle che i tool di beautifying automatici rovinano.

    <!-- Do this -->
<p>
  There was a farmer who had a dog, and <em>Bingo</em> was his name-o...
</p>

<!-- Don't do this -->
<p>
  There was a farmer who had a dog, and
  <em>Bingo</em>
  was his name-o...
</p>

    Per gli inline elements “strutturali” (ad es. <img>, <button>), e’ meglio metterli su una riga separata come i block elements. In effetti, questi elementi spesso vogliono essere stilizzati come block elements!

    <!-- Do this -->
<p>
  Questa e' una foto di un cane:
  <img src="puppy.png" alt="un cucciolo">
</p>

<!-- Don't do this -->
<p>
  Questa e' una foto di un cane: <img src="puppy.png" alt="un cucciolo">
</p>

  • Gli element attributes dovrebbero essere scritti sulla stessa riga del tag a cui si applicano—anche se il valore dell’attribute sembra “lungo” (come un URL).

    <!-- Do this -->
<a href="https://info340.github.io/really/long/path/to/content">Link</a>

<!-- Don't do this -->
<a
  href="https://info340.github.io/really/long/path/to/content">Link</a>

    Nota che non e’ HTML valido inserire whitespace, incluse linee di interruzione, attorno al = quando definisci un attribute.

    L’unica eccezione a questa linea guida e’ quando hai molti attributes per un singolo elemento. In quel caso, e’ accettabile mettere ogni attribute su una riga separata, indentata di 1 livello. Anche se succede raramente in HTML puro, e’ molto comune quando specifichi props per un componente React (e rende piu’ facile modificare quelle props).

    <!-- Do this -->
<input
  type="text"
  id="user-input-field"
  name="user-input-field"
  value=""
  placeholder="Write something here!"
  class="long-form-input">

  • Scrivi tutto il contenuto testuale di un singolo elemento su una singola riga di codice; non inserire manualmente line break nel plain text. Questo ti permette di modificare quel contenuto in seguito senza dover riformattare tutto. Non preoccuparti della lunghezza della riga di codice. Puoi usare la funzionalita’ “word wrap” del tuo editor per evitare lo scrolling orizzontale.

    <!-- Do this -->
<!-- This *single line of code* can be made to wrap in your editor -->
<p>
  Lorem, ipsum dolor sit amet consectetur adipisicing elit. Consectetur est necessitatibus, rerum atque officiis doloremque porro similique molestias fugit, a repellendus fuga natus, tempora impedit. Dolore repellendus itaque soluta est ad modi corrupti quibusdam tenetur architecto nesciunt harum ipsa consectetur ullam unde, quos sit asperiores corporis vitae pariatur expedita non?
</p>

<!-- Don't do this -->
<p>
  Lorem, ipsum dolor sit amet consectetur adipisicing elit.
  Consectetur est necessitatibus, rerum atque officiis doloremque
  porro similique molestias fugit, a repellendus fuga natus,
  tempora impedit. Dolore repellendus itaque soluta est ad modi
  corrupti quibusdam tenetur architecto nesciunt harum ipsa
  consectetur ullam unde, quos sit asperiores corporis vitae
  pariatur expedita non?
</p>

Elementi specifici

  • Evita elementi di formattazione non semantici (ad es. <i>, <b>), perche’ non aggiungono significato al contenuto.

  • Evita di usare l’elemento <br> per gli a capo. Se stai spezzando una riga di testo, di solito e’ perche’ stai definendo un nuovo paragraph, quindi dovresti usare un altro <p>. Se non c’e’ un significato semantico per il line break, non inserirlo! Se devi regolare la spaziatura tra paragraph (ad es. vuoi una spaziatura singola invece che doppia), usa il CSS per regolare margin o padding.

    <!-- Do this -->
<p>
  Lorem, ipsum dolor sit amet consectetur adipisicing elit.
</p>
<p>
  Consectetur est necessitatibus, rerum atque officiis doloremque porro similique molestias fugit, a repellendus fuga natus, tempora impedit.
</p>

<!-- Don't do this -->
<p>
  Lorem, ipsum dolor sit amet consectetur adipisicing elit.
  <br />
  Consectetur est necessitatibus, rerum atque officiis doloremque porro similique molestias fugit, a repellendus fuga natus, tempora impedit.
<p>

    L’unico motivo che mi viene in mente per usare semanticamente un elemento <br> e’ nella poesia:

    <!-- Do this -->
<div>
  Roses are red, <br>
  Violets are blue. <br>
  Unexpected '}' <br>
  on line 32.
</div>

Commenti in HTML

  • La maggior parte del contenuto HTML non ha bisogno di commenti; dovrebbe essere comprensibile solo dall’indentazione e dal contenuto. Se il tuo codice ha bisogno di piu’ organizzazione, usa semantic elements (ad es. <section>) o class names per aiutarne la struttura (ad es. <div class="first-post">).

21.2 CSS Guidelines

Un buon consiglio per scrivere CSS e’ concentrarsi sulla quantita’ “minima” di CSS necessaria per ottenere l’effetto desiderato. Lavora con il browser e i suoi default, invece di cercare di dominarli. Forse sorprendentemente, rimuovere rules o properties e’ spesso una soluzione migliore di aggiungere ulteriore styling.

21.2.1 Spacing

  • Quando scrivi CSS rules, metti la { di una rule sulla stessa riga del selector e indenta le properties della rule di una quantita’ consistente. Metti uno spazio dopo : in una rule, ma non prima. Le rules dovrebbero apparire cosi’:

    selector {
    property: value;
    property: value;
}

    Nota che puoi usare VS Code per “indentare” automaticamente il codice con il comando format document.

21.2.2 Selectors

  • Usa selectors che siano solo quanto basta specifici.

    /* Do this */
li.selected {} /* selected list items, not selected paragraphs */

/* Don't do this */
div p {} /* paragraphs inside of divs (are there other paragraphs? */

/* Don't do this */
body p {} /* no paragraphs outside of the body, so redundant */

    (E’ difficile mostrare esempi, perche’ la specificita’ richiesta dipende dalla situazione!)

    La maggior parte dei selectors avra’ 1-3 “pezzi” (connessi da selezione discendente o compound). Se ti trovi a usarne di piu’, chiediti se c’e’ un modo piu’ semplice di scrivere il selector—o dai semplicemente all’elemento una class e selezionalo direttamente!

  • Usa class o element selectors invece di id selectors.

    /* Do this */
.side-nav {}

/* Don't do this */
#side-nav {}

  • Se devi usare un selector id, non aggiungere specificita’ extra prima degli id selectors. Un id deve essere unico, quindi non devi distinguere tra piu’ elementi con lo stesso id.

    /* Don't do this */
nav#side-nav {}

  • Non usare la keyword !important. Scrivi invece una rule sufficientemente specifica!

    /* Don't do this */
.error {
  color: red !important;
}

/* Do this */
p.alert.error {
  color: red;
}

21.2.3 Class Names

  • Usa class names descrittivi. Le class dovrebbero descrivere cosa o perche’ quello styling viene applicato.

    /* Do this */
.side-nav {}

/* Do this */
.alert-warning {}

/* Don't do this */
.thingy {}

/* Don't do this */
.s1 {}

/* Don't do this */
.p {} /* use an element selector instead! */

    Nota che le class possono essere definite semanticamente (ad es. .avatar-icon, .comment-form) oppure modularmente (ad es. .font-large, .bg-secondary). Entrambi gli approcci sono accettabili in questa guida, ma cerca di essere consistente.

    Usare uno naming schema come BEM e’ anche accettabile.

  • Usa i trattini come delimiters per i class names.

    /* Do this */
.side-nav {}

/* Do this */
.alert-warning {}

    Alcuni framework possono rendere sensato usare il camelCasing per i class names (cosi’ i class names sono anche valid JavaScript identifiers). In quel caso, sii consistente con i delimiters. Questo rendera’ piu’ facile scrivere e modificare il codice.

    /* Don't do this */
.sideNav {}
.alert-warning {}

21.2.4 Proprieta’ specifiche

  • Evita di usare float; usa flexboxes o grids per il positioning. E’ raro avere davvero contenuto “floating” (anche se puo’ succedere con immagini inserite in grandi articoli testuali).

  • Evita di usare position: absolute. Questo produce layout non responsive o non accessibili su piu’ dispositivi. Lavora invece con il layout del browser usando layout relativi.

21.2.5 Responsive CSS

  • Usa un approccio mobile-first allo styling. Questo significa che le regole generali vanno in alto e si applicano ai device mobile, e poi usi media queries per specificare modifiche di stile per display piu’ grandi.

  • Le media queries controllano solo la size minima:

    /* Do this */
@media (min-width: 768px) {}

/* Do not do this */
@media (min-width: 768px) and (max-width: 1092px) {}

/* Do not do this */
@media (max-width: 768px) {}

  • Limitare le styling rules a screen non e’ necessario (a meno che tu non voglia esplicitamente che siano diverse dalle regole di stampa)

    /* Do this */
@media (min-width: 768px) {}

/* Do not need to do this */
@media screen and (min-width: 768px) {}

  • Le media queries dovrebbero essere usate per modificare lo styling mobile, non per sostituirlo completamente. Non “resettare” tutte le rules dello styling mobile, ma sovrascrivi e aggiungi solo le poche rules necessarie per rendere la pagina realmente responsive. La tua pagina dovrebbe avere uno style/theme simile a prescindere dal device usato per visualizzare il contenuto!

21.2.6 Commenti in CSS

  • Non hai bisogno di molti commenti per il codice CSS; le rules dovrebbero essere auto-esplicative perche’ hai usato class names descrittivi.

  • Usa i commenti per organizzare o “segnalare” il codice, raggruppando le rules nel file .css.

    /* Do this */

/* navbar */
nav {}
nav li {}
.tab-selected {}

/* main content */
section {}
p {}
img.small {}

21.3 JavaScript Guidelines

21.3.1 Variables

  • Dichiara le variables con const o let. In generale, usa const per tutte le variables, e poi let solo se scopri di dover riassegnare la variable in seguito (cosa molto meno comune di quanto pensi)!

    Non usare var per dichiarare variables (questo evita di inquinare il global scope).

  • I nomi delle variables si scrivono in camelCaseFormat (capitalizzando la prima lettera di ogni “word” dopo la prima). Non iniziare i nomi delle variables con una lettera maiuscola.

    /* Do this */
const myName = "Joel Ross"

/* Do not do this */
const my_name = "Joel Ross"

/* Do not do this */
const MyName = "Joel Ross"

    Puoi usare tutte lettere maiuscole per le global constants. Nota che solo perche’ una variable e’ dichiarata const non significa che sia una “global constant”. Usa tutte lettere maiuscole per nominare valori specificati esternamente al funzionamento del programma; PI, WA_TAX_RATE, SCREEN_WIDTH sono tutte global constants.

  • Usa nomi di variable descrittivi. I nomi delle variables dovrebbero indicare a cosa si riferisce il value. Non usare nomi come stuff, thing o x perche’ non aiutano nessuno a capire il codice.

  • Dai nome ad arrays e collections usando nomi plurali. In alternativa, potresti trovare utile nominare strings, arrays, objects e functions in base al loro data type per aiutarti a ricordare (in particolare se i plurali in inglese sono difficili).

    /* Do this */
const names = ["john", "paul", "george", "ringo"];

/* Do this */
const dogArray = ["fido", "spot", "rover"]; //array of dog names
const dogObj  = {name: 'Fido', breed: 'mutt'}; //single dog object

/* Do not do this */
const dog = ["fido", "spot", "rover"]; //an array named by a singular noun

21.3.2 Functions

  • Usa function declarations invece di function expressions quando definisci functions—soprattutto le functions “top-level”. Questo aiuta a distinguere tra functions e variables, rendendo il codice piu’ facile da leggere e seguire (anche se richiede piu’ digitazione).

    /* Do this */
function sum(a, b) {
  return a + b;
}

/* Do not do this */
const sum = function(a, b) {
  return a + b;
}

/* Do not do this */
const sum = (a, b) => a + b;

  • Per le inline functions (come le anonymous callback functions), usa la arrow notation. Questo rende il codice piu’ conciso e aiuta a evitare problemi di scope.

    /* Do this */
const transformed = array.map((item) => {
  //...
})

/* Do not do this */
const transformed = array.map(function(item) {
  //...
})

    Quando usi le arrow functions, metti sempre le () attorno alla lista degli argomenti. Questo chiarisce che si tratta di una function e rendera’ piu’ facile aggiungere argomenti in seguito.

    /* Do this */
const transformed = array.map((item) => {
  //...
})

/* Do not do this */
const transformed = array.map(item => {
  //...
})

    In generale, evita di usare concise body arrow functions. Includere il block esplicito ({}) rende piu’ facile leggere la function e anche modificarla e fare debug, perche’ puoi aggiungere statements facilmente. Sembra piu’ codice da digitare, ma e’ uno style migliore.

    /* Do this */
const exclaimed = stringArray.map((aString) => {
  return aString + "!!" //add exclamation points
})

/* Do not do this */
const exclaimed = stringArray.map((aString) => aString + "!!")

  • Le functions dovrebbero essere definite per essere short e reusable. Usa pure functions quando possibile. Questo significa che evitano side effects—non assegnano o modificano variables non locali.

21.3.3 Commenti in JavaScript

“Comments are always failures.” - Robert Martin

In generale, il codice ben scritto si auto-documenta senza bisogno di commenti aggiuntivi. Functions ben nominate con scope piccoli indicano chiaramente cosa fanno senza bisogno di ulteriori note. Se ti accorgi che serve un commento per spiegare cosa fa il codice, probabilmente dovresti riscriverlo per renderlo piu’ leggibile. I commenti dovrebbero fornire informazioni sull’intent del codice (perche’ e’ stato incluso), non sul behavior del codice (cosa fa).

Questo significa che in generale non dovresti aver bisogno di molti commenti nel codice. Usa i commenti come ultima risorsa per chiarire il behavior del codice o per comunicare con altre persone.

Non lasciare grandi blocchi di codice commentato nelle versioni production (finali)—mantieni il codice chiaro senza costringere chi legge a scorrere o ignorare tanto codice inutilizzato.

21.3.4 Altre linee guida JavaScript

  • Usa confronti di uguaglianza strict (=== e !==) invece di confronti di uguaglianza regolari (== e !=).

  • Minimizza l’uso di statements console.log() nelle versioni production (finali). Rallentano il programma, inquinano lo spazio dei log e possono essere fonte di leak di informazioni e sicurezza. Usa molti console.log() quando fai debugging, ma rimuovili quando hai risolto i bug.

21.4 React Guidelines

21.4.1 Components

  • Usa Component functions (e hooks). Non definire componenti come classi.

  • Usa function declarations invece di function expressions quando definisci Component functions. Questo aiuta la leggibilita’.

    /* Do this */
function Card(props) {
  //...
}

/* Do not do this */
const Card = (props) => {
  //...
}

  • Le Component functions si nominano con nouns (cosa sono), non con verbi (cosa fanno). Nominale come faresti con classes in Java o altri linguaggi OOP.

    /* Do this - what the content is */
function EntryForm(props)
  //...
}

/* Do not do this - what it does */
function UpdateData(props) {
  //...
}

    Dai ai Components nomi descrittivi che indicano che “kind” di elemento sono, cosi’ non c’e’ confusione tra nouns e verbi.

    /* Do this - the component is a form */
function FilterForm(props)
  //...
}

/* Do not do this - unclear what the component is */
function Filter(props) {
  //...
}

  • I Components sono sempre scritti come top level functions. Non definire mai un componente dentro un altro!

    /* Do this */
function Parent(props)
  //...
}

function Child(props)
  //...
}


/* Do not do this */
function Parent(props) {

  function Child(props) {
    //...
  }

  //...
}

  • L’argomento di un Component si chiama sempre props (con la s alla fine!).

    E’ accettabile usare l’object destructuring nell’argomento di una Component function. Ricorda di includere le {}—un Component accetta un solo argomento!

    /* Acceptable */
function SongCard(props)
  //...
}

/* Also acceptable */
function SongCard({artist, title, album}) {
  //...
}

  • Quando specifichi props per un Component (o attributes per un elemento HTML), non mettere spazi attorno a = — scrivilo come in HTML!

    /* Do this */
<Card value={dataItem} />

/* Do not do this */
<Card value = {dataItem} />

  • Organizza i Components in moduli separati (file .js). Puoi includere piu’ componenti correlati (ad es. un Card e un CardList) nello stesso modulo. E’ una buona idea tenere questi file insieme in una cartella components distinta nel tuo source code.

21.4.2 Mapping Data

  • Non dichiarare functions come la callback di .map() inline dentro lo statement return di un Component (o di qualsiasi altra function). Usa statements multipli quando scrivi codice!

    Invece, dichiara i values mappati come una variable separata che puo’ essere inclusa nel DOM restituito. Questo rende il codice piu’ facile da leggere, modificare e fare debug, perche’ puoi aggiungere ulteriore processing o ispezioni dopo la chiamata .map().

    /* Do this */
function App(props) {
  //map the data into <Card> elements
  const cardElemArray = data.map((dataItem) => {
    return <Card value={dataItem} key={dataItem.id} />
  })

  return (
    <div>
      {cardElemArray}
    </div>
  )
}

/* Do not do this */
function App(props) {
  //Questo e' tutto un unico statement (linea) di codice! Non farlo.
  return (
    <div>
      {
        data.map((dataItem) => {
          return <Card value={dataItem} key={dataItem.id} />
        })
      }
    </div>
  )
}

21.4.3 State

  • Usa sempre const quando dichiari state variables (dato che non le riassegni comunque)!

    /* Do this */
const [data, setData] = useState([]);

/* Do not do this */
let [data, setData] = useState([]);

  • Sempre nomina la function “state setter” a partire dal nome della state variable (set______):

    /* Do this */
const [data, setData] = useState([]);
const [currentsong, setCurrentSong] = useState({});

/* Do not do this */
const [nowPlaying, changeSong] = useState({});
const [lastTrack, setLast] = useState({}); //use the full variable name instead!

  • Non duplicare i data nello state. E’ cattivo style e causa bug. Mantieni lo state minimale.

    Non definire state variables per values che possono essere calcolati da altre variables.

    /* Do this */
const [myArray, setMyArray] = useState([]);
const arrayLength = myArray.length; //computed from state, but not a state variable

/* Do not do this */
const [myArray, setMyArray] = useState([]);
const [arrayLength, setArrayLength] = useState(0); //duplicated data

    Non far salvare gli stessi data nello state di piu’ componenti. Invece, fai lift dello state al livello appropriato e passa i data verso il basso come prop.

    /* Do this */
function App(props) {
  const [data, setData] = useState([]);

  return (
    {/* pass down the data as a prop */}
    <CardList data={data} />
  )
}


/* Do not do this */
function App(props) {
  const [data, setData] = useState([]);

  return (
    <CardList data={data} />
  )
}

function CardList(props) {
  const [data, setData] = useState(props.data);
  //...
}

  • Non passare le state setter functions come callback ai child components. Invece, crea una callback function separata che possa chiamare lo state setter. Questo rende piu’ facile aggiungere processing e logica di controllo quando aggiorni lo state, oltre a seguire il Principle of Least Knowledge.

    /* Do this */
function App(props) {
  const [data, setData] = useState([]);

  //a function to update the data
  const updateData = (newData) => {
    //can do additional processing here
    setData(newData); //update the state
  }

  return (
    {/* pass down the callback function as a prop */}
    <UpdateForm data={data} updateCallback={updateData} />
  )
}


/* Do not do this */
function App(props) {
  const [data, setData] = useState([]);

  return (
    {/* Do not pass the state setter directly! */}
    <UpdateForm data={data} updateCallback={setData} />
  )
}

21.4.4 Events e Forms

  • Tutti i form dovrebbero essere controlled, gestendo il valore inserito tramite lo state del Component (invece dello state dell’elemento HTML). Questo aiuta a evitare la duplicazione dei data, seguendo il Don’t Repeat Yourself (DRY) Principle.

    /* Do this */
function ExampleForm(props) {
  const [inputValue, setInputValue] = useState('')

  const handleChange = (event) => {
    let newValue = event.target.value
    setInputValue(newValue);
  }

  return (
    <input type="text" onChange={handleChange} value={inputValue} />
  )
}

21.5 Linee guida varie

  • Dai a tutti i file e folder (source code o altro) nomi con tutte lettere minuscole. Questo mantiene consistenza ed evita bug tra sistemi operativi.

  • Non includere spazi () nei nomi dei file, in particolare per i media assets—non avere un file chiamato my puppy.png. Sostituisci lo spazio con un carattere alternativo, come _, - o +. I whitespace characters devono essere codificati in modo speciale per gli URI, il che puo’ causare problemi e di leggibilita’.
