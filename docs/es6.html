Chapter 15 Sintassi ES6+

Come discusso nel Chapter 10, la specifica ECMAScript per il linguaggio JavaScript ha attraversato diverse versioni, ognuna delle quali ha aggiunto nuova sintassi e nuove feature per rendere il linguaggio piu’ potente, espressivo o piu’ facile da usare.

Dopo che il linguaggio JavaScript fu scritto inizialmente nel 1995 e poi iterato fino alla fine degli anni ‘90, rimase per lo piu’ consistente (con piccoli cambiamenti) per circa un decennio—fino al 2009 con il rilascio di JavaScript v5. La versione successiva importante non usci’ fino al 2015—anche se ufficialmente chiamata “ECMAScript 2015”, la maggior parte dei developer la chiama con il nome di lavoro “ES6” (ad es. versione 6 del linguaggio). ES6 ha introdotto molte feature sintattiche notevoli e utili—inclusi let e const! Da allora, il linguaggio e’ stato aggiornato annualmente, con ogni versione chiamata secondo l’anno di rilascio (ES 2016 e’ uscito nel 2016, ES 2017 nel 2017, ecc).

Questo capitolo introduce alcune delle feature piu’ notevoli e utili introdotte in ES6 e nelle versioni successive di JavaScript—in particolare quelle necessarie quando si usa il framework React (discusso nei capitoli seguenti).

A questo punto ES6 e’ sufficientemente maturo da essere quasi completamente supportato dai browser moderni, con l’eccezione significativa di Internet Explorer. Tuttavia, l’interpreter JavaScript nei browser piu’ vecchi (e in IE) non riconosce la nuova sintassi introdotta in questa versione. Invece, dovresti convertire quel codice in equivalente ES5 (o precedente), che puo’ essere interpretato. Un modo per farlo e’ usare il compiler Babel, che “transpila” il codice JavaScript da una versione a un’altra. Il capitolo successivo discute come fare questo tipo di transpiling con React (spoiler: e’ fatto automaticamente dai build tools forniti), ma e’ anche possibile installare e usare Babel da soli.

15.1 ES6+ Syntax Features

Syntactic Sugar causes cancer of the semicolon - Alan Perlis

Molte delle feature ES6 sono syntactic shortcuts—forniscono sintassi per scrivere functions e operazioni in modo piu’ conciso. Queste feature non sono necessarie per scrivere JavaScript, ma possono rendere il codice piu’ facile da scrivere e da leggere (una volta appresa la sintassi), e sono considerate il modo “normale” di scrivere JavaScript moderno. Questa sezione discute alcune delle feature sintattiche piu’ comuni.

Arrow Functions

Come descritto nel Chapter 11, JavaScript permette di definire functions come anonymous values:

const sayHello = function(name) {
  return 'Ciao '+name;
}

Come hai visto, l’uso di anonymous functions e’ incredibilmente comune in JavaScript, soprattutto quando usate come anonymous callbacks. Proprio per questo, ES6 ha introdotto una sintassi piu’ semplice e concisa per specificare rapidamente anonymous functions. Anche se ufficialmente chiamate lambda functions, sono piu’ comunemente note come arrow functions, per via dell’uso del simbolo “arrow” =>:

const sayHello = (name) => {
    return 'Ciao '+name;
}

Per trasformare una anonymous function in arrow function, basta rimuovere la keyword function dalla definizione e mettere un arrow => tra la lista dei parametri e il block (indicando che la lista dei parametri “va verso” il block successivo). Questo fa risparmiare qualche carattere quando digiti!

E’ anche possibile omettere le () attorno alla lista degli argomenti in una arrow function:

//puoi omettere le parentesi attorno all'argomento -- ma non farlo!
const sayHello = name => {
    return 'Ciao '+name;
}

//le parentesi sono richieste se non ci sono argomenti
const printHello = () => {
    console.log('Ciao mondo');
}

Tuttavia, questo rende la sintassi piu’ difficile da leggere (leggendo il testo, sembra una dichiarazione di variable finche’ non arrivi al =>). Quindi dovresti includere sempre le parentesi () nella lista dei parametri: aiuta la leggibilita’ e rende piu’ facile aggiungere parametri in seguito—rispettando i requisiti di good code style.

Le arrow functions sono particolarmente comode quando usi anonymous callback functions (function literals):

const array = [1,2,3]; //un array con cui lavorare

//sintassi di dichiarazione della function
array.map(function(num) {
  return num * 2; //moltiplica ogni item per 2
});

//sintassi arrow function
array.map((num) => {
  return num * 2; //moltiplica ogni item per 2
});

Usare una arrow function rende leggermente piu’ ovvio che la function prende un input e poi produce (=>) un output—cosi’ puoi concentrarti sul metodo di interesse (come .map()) invece che sulla callback. Tuttavia, rende meno ovvio che l’argomento di .map() e’ una function finche’ non ti abitui a riconoscere la freccia =>.

Le arrow functions sono lexically scoped, cioe’ il value della keyword this viene mantenuto anche dentro la function. Questo puo’ essere importante in alcuni contesti (come le classes) ed e’ uno dei motivi per cui e’ meglio usare arrow functions quando definisci callback functions).

Per functions molto semplici, le arrow functions possono essere scritte in modo ancora piu’ compatto: se il body della function ha solo uno statement, puoi omettere le {} del block e anche il return. Questa si chiama concise body. La function restituira’ il risultato del singolo statement, che sara’ un’espressione o undefined. Per esempio:

//sintassi di dichiarazione della function
function makeFullName(first, last) {
  return first + " " + last;
}

//sintassi arrow function a block body
const makeFullName = (first, last) => {
  return first + " " + last;
}

//sintassi arrow function a concise body
const makeFullName = (first, last) => first + " " + last;

Nota che se l’espressione di una concise-body arrow function restituisce undefined (ad es. da un console.log), allora anche la arrow function restituira’ undefined.

//sintassi arrow function a concise body
const sayHello = (name) => console.log("Ciao" + name);

//questo e' identico all'uso della block body syntax. Si, restituire il risultato di
//console.log() e' strano
const sayHello = (name) => {
  return console.log("Ciao" + name);
}

Usare la concise body synatx potrebbe farti risparmiare qualche carattere (e anche line break!). Tuttavia, puo’ rendere la function piu’ difficile da leggere—soprattutto quando ha piu’ di un argomento—e piu’ difficile da riconoscere come function perche’ il block non e’ evidente. Le concise body functions sono anche piu’ difficili da modificare—possono avere solo uno statement, quindi se vuoi che la function faccia piu’ di una cosa—incluso usare un loop o uno if statment—devi comunque convertirla in block body. (Cercare di rendere quell’unica espressione molto complessa e’ anche poor style). E dato che non puoi aggiungere righe, e’ piu’ difficile fare debug perche’ non puoi aggiungere console.log o altri passi di debugging. Per questo, dovresti quasi mai usare la concise-body arrow function syntax se non per callback molto semplici—e’ sempre meglio usare la block body syntax.

In generale, le arrow functions (in particolare le block body arrow functions) sono un clean syntactic shortcut che e’ il modo “normale” di scrivere anonymous callback functions—le vedrai in tanti esempi e nel codice professionale. Le “top-level” functions dovrebbero essere scritte con la sintassi regolare di function declaration, ma le callback functions dovrebbero sempre essere scritte usando block-body arrow functions.

Destructuring

ES6 ha introdotto anche i destructing assignments, che ti permettono di assegnare ogni elemento di un array (o ogni property di un object) a variables separate in un’unica operazione. Si chiama “destructuring” perche’ stai “unpacking” i values di un array in un insieme di variables.

Per fare destructuring di un array in JavaScript, scrivi le variables a cui vuoi assegnare i values dentro parentesi quadre [] sul left-hand side dell’assegnazione—quasi come se stessi assegnando a un array!

const myArray = [1, 2, 3];
const [x, y, z] = myArray; //assegna gli elementi di myArray a `x`, `y`, `z` rispettivamente
console.log(x); //=> 1;
console.log(y); //=> 2;
console.log(z); //=> 3;

Quando fai destructuring di un array, i values sono assegnati in ordine—il value all’index 0 va alla prima variable, il value all’index 1 alla seconda, ecc.

E’ possibile che la “lista di variables” e la size dell’array destructured non corrispondano. Se nell’array ci sono piu’ values che variables, i values extra non verranno assegnati (saranno ignorati). Se ci sono piu’ variables che values, le variables in eccesso saranno undefined (perche’ il value in quella posizione dell’array e’ undefined).

const bigArray = [1, 2, 3, 4, 5];
const [a, b] = bigArray; //solo i primi 2 elementi verranno assegnati
console.log(a); //=> 1;
console.log(b); //=> 2;
//gli altri elementi non sono assegnati

const smallArray = [1, 2]
const [x, y, z] = smallArray;
console.log(x); //=> 1;
console.log(y); //=> 2;
console.log(z); //=> undefined; (perche' smallArray[2] e' undefined!)

E’ anche possibile fare destructuring di objects usando una sintassi simile, tranne per il fatto che metti le variables a cui vuoi assegnare dentro {} sul left-hand side dell’assegnazione—come se stessi assegnando a un object!

const myObject = {a: 1, b: 2, c: 3};
const {a, b, c} = myObject;
console.log(a); //=> 1; myObject.a
console.log(b); //=> 2; myObject.b
console.log(c); //=> 3; myObject.c

Nota che quando fai destructuring di un object, le variables specificate sul lato sinistro dell’assegnazione ricevono il value della property corrispondente nell’object. Quindi in const {a, b, c} = myObject, la a viene assegnata a myObject.a, la b a myObject.b, ecc. L’ordine degli elementi su entrambi i lati dell’equazione non conta, perche’ le object properties sono unordered!

const myObject = {a: 1, b: 2, c: 3};
const {c, b, a} = myObject; //"ordine" delle variables non importa
console.log(a); //=> 1
console.log(b); //=> 2;
console.log(c); //=> 3;

Esiste anche sintassi per assegnare properties dell’object a nomi di variable diversi (vedi la documentazione sul destructuring per dettagli).

Come per il destructuring di array, la lista di variables e il numero di object properties non devono per forza coincidere. Le properties non abbinate non verranno assegnate a variables, e le variables senza property corrispondenti resteranno undefined.

Il destructuring e’ uno syntactic shortcut utile per fare assegnazioni multiple. In React e’ spesso usato per creare variables locali, cosi’ non devi scrivere continuamente object.property:

//una function che si aspetta un object come argomento
function getBMI(personObject) {
  const {height, weight} = personObject; //destructuring in variables locali
  //evita di dover riferirsi a `personObject.height`

  return 703*weight/(height*height); //calcola e restituisce il value
}

const person = {name: 'Ada', height: 64, weight: 135} //una persona di esempio
const adaBMI = getBMI(person); //passa l'object

In effetti, questo comportamento e’ cosi’ comune che e’ possibile fare destructuring dei function parameters nella function declaration stessa! Per farlo, sostituisci il nome dell’object nella lista dei parametri con l’insieme delle variables destructured (cio’ che starebbe sul left-hand side del destructuring):

//una function con l'argomento object destructured
//nota il `{} nella lista degli argomenti`
function getBMI( {height, weight} ) { //chiama implicitamente `{height, weight} = personObject`

  return 703*weight/(height*height); //calcola e restituisce il value
}

const person = {name: 'Ada', height: 64, weight: 135} //una persona di esempio
const adaBMI = getBMI(person); //passa l'object

Per essere chiari: nel secondo esempio, la function getBMI() si aspetta solo 1 argomento—quell’unico argomento viene semplicemente destructured in due variables quando la function viene chiamata.

Questa sintassi puo’ farti risparmiare una riga di codice, ma puo’ anche rendere piu’ facile fraintendere la function e pensare che prenda piu’ argomenti di quanti ne prenda davvero (se ti sfuggono le {} nella lista dei parametri). D’altro canto, questa sintassi rende chiaro esattamente quali properties deve avere l’object atteso—questa function definisce chiaramente la sua API: il “person object” deve avere height e weight (e non solo, ad es., name o age). Come per ogni sintassi alternativa, ci sono trade-off in termini di code style e leggibilita’. Tuttavia, questa forma di sintassi e’ spesso usata in React (dove molte functions si aspettano un singolo object, ma devi accedere alle properties individuali).

Spreading

Introdotto in ES6 e poi ampliato in ES 2018, lo spread operator (l’ellissi ...) ti permette di riferirti a “gli elementi di” un array o object esistente quando dichiari un nuovo value. “Espande” i contenuti dell’array o dell’object nel nuovo value. Lo scrivi come unary operator, mettendo i tre punti davanti al value che vuoi espandere:

const originalArray = ['a', 'b', 'c', 'd'];

//il nuovo array contiene "gli elementi di" `originalArray`
const newArray = [...originalArray, 'e', 'f'];

console.log(newArray) //['a', 'b', 'c', 'd', 'e', 'f']

In questo esempio il newArray contiene tutti gli elementi di originalArray, oltre a elementi aggiuntivi. E’ particolarmente utile quando vuoi creare una “copy” di un array o object, con o senza values aggiuntivi.

//assegnare un array a una variable diversa non crea un nuovo array!
const myArray = ['a', 'b', 'c', 'd'];
const secondArray = myArray; //solo una nuova etichetta per lo stesso value
myArray[0] = 'Z'; //modificare la variable originale modifica anche la seconda variable
console.log(secondArray[0]) //'Z'

//usa lo spread per creare una vera copia che e' un array diverso
const clonedArray = [...myArray];
myArray[1] = 'Q'; //modifica la variable originale
console.log(clonedArray[1]); //'b'; clonedArray e' diverso

Lo spread operator puo’ essere usato anche per creare nuovi objects; fare spread di un object espande sia le property keys che i values:

const person = {name: 'Ada', height: 64, weight: 135}

const copyOfPerson = {...person}; //clona l'object
console.log(copyOfPerson); // {name: 'Ada', height: 64, weight: 135}

//tutte le properties sono "spalmate" nel nuovo object
const personWithHat = {hat: 'bowler', ...person}
console.log(person); //ha properties 'name', 'height', 'weight'
console.log(personWithHat); //ha properties 'hat', 'name', 'height', 'weight'

Quando combinato con il destructuring, lo spread operator puo’ riferirsi anche al “resto” degli elementi di un array o object che non sono stati assegnati a specifiche variables. In questa situazione spesso si parla di rest operator.

const dimensions = [10, 20, 30, 40];
//i values extra sono "spalmati" negli slot di destructuring
const [width, height, ...others] = dimensions
console.log(width);  //=> 10
console.log(height); //=> 20
console.log(others);   //=> [30, 40]; il resto dei values!

Infine, lo spread operator puo’ essere usato per specificare che una function puo’ accettare un numero variabile di argomenti e raccogliere tutti questi values in un singolo array. Lo fai includendo lo spreading operator nella function declaration. Gli argomenti verranno raccolti in quella singola variable—quasi il contrario del destructuring! (E si’, il fatto che lo “spread” operator “raccolga” i values e’ un po’ confuso).

//una function che stampa tutti gli arguments
function printArgs(...argArray){
    //tutti gli argomenti saranno raggruppati in un singolo array `args`
    for(const arg of argArray) {
      console.log(arg); //puoi stamparli tutti, non importa quanti!
    }
}

printArgs('a', 'b', 'c'); //=> "a" "b" "c"
printArgs(1, 2, 3, 4, 5, 6); //=> "1" "2" "3" "4" "5" "6"


//una function che somma tutti gli argomenti (non importa quanti!)
function sum(...numbers) {
    //numbers e' un array, quindi possiamo usare `reduce()`!
    const total = numbers.reduce((runningTotal, num) => {
        return runningTotal + num; //nuovo totale
    }, 0); //parti da 0

    return total;

    //oppure su una riga con una concise arrow function (non bello!)
    return numbers.reduce((total, n) => total+n);
}

sum(3 ,4, 3); // => 10
sum(10, 20, 30, 40); // => 100

Lo spread operator non e’ usato spesso in questo modo con React, ma e’ utile saperlo!

Template Strings

In ES6, puoi dichiarare Strings che contengono espressioni embedded, permettendoti di “iniettare” un’espressione direttamente in una string (invece di concatenare la String con quell’espressione). Queste si chiamano template strings (o template literals). Le template strings si scrivono con i back tick (``) invece delle quotes, con le espressioni inserite dentro il token ${}:

const name = 'mondo';
const greeting = `Ciao, ${name}!`; //template string
console.log(greeting); //=> "Ciao, mondo!"

Le template strings possono anche includere line break, permettendoti di creare stringhe multi-linea!

Nota che puoi mettere qualsiasi espressione dentro il token ${}; tuttavia, e’ best practice mantenere l’espressione il piu’ semplice possibile (ad es. usando una variable locale) per supportare la leggibilita’:

const name = 'mondo';

//saluto con capitalizzazione. Non farlo cosi'!
const greeting = `Ciao, ${name.substr(0,1).toUpperCase() + name.substr(1)}!`
console.log(greeting); //=> "Ciao, Mondo!";

//fai invece cosi'!
const capitalizedName = name.substr(0,1).toUpperCase() + name.substr(1);
const greeting = `Ciao, ${capitalizedName}`
console.log(greeting); //=> "Ciao, Mondo!";

Un altro esempio:

//la function si aspetta un nome, un animale e un verbo
function makeExcuse(name, animal, verb) {
  const email = `Ciao Professore ${name},

Per favore scusi il mio compito mancante, perche' il mio ${animal} l'ha mangiato.

${verb} a presto,
Uno Studente`;

  return email;
}

const excuse = makeExcuse('Ross', 'Lemur', 'Smell');

15.2 Modules

Finora hai probabilmente scritto tutto il codice JavaScript in un singolo file di script (ad es. index.js), anche se hai incluso altre librerie come Bootstrap o JQuery tramite tag <script> aggiuntivi. Ma man mano che le application diventano piu’ grandi e complesse, questo singolo file puo’ diventare rapidamente ingestibile con centinaia o migliaia di righe di codice che implementano decine di feature. File cosi’ grandi diventano difficili da leggere e fare debug (“dov’era definita quella function?”), possono introdurre problemi con il global namespace condiviso (“ho gia’ dichiarato una variable user?”), e in generale mescolano il codice in modo che viola il principio di Separation of Concerns.

La soluzione a questo problema e’ dividere il codice dell’applicazione in modules (script) separati, ciascuno responsabile di un pezzo specifico di funzionalita’. E invece di caricare ogni module con un tag <script> separato (con possibili problemi di ordering e dependency, oltre a inquinare il global namespace), puoi definire ogni module come script autonomo che “importa” esplicitamente le functions e le variables di cui ha bisogno da altri moduli. Questo ti permette di organizzare meglio il programma man mano che cresce.

Anche se separare il codice in moduli e’ comune in ambiente Node.js, ES6 aggiunge una sintassi che permette di trattare i singoli file .js come moduli che possono comunicare tra loro. Questi sono noti come ES6 Modules.

Un browser carica i moduli in modo diverso dagli script “regolari”; per caricare codice come module, specifichi l’attribute type="module" nel tag <script> di quel module:

<!-- carica script che possono importare da altri moduli -->
<script type="module" src="path/to/firstModule.js"></script>
<script type="module" src="path/to/secondModule.js"></script>

Per motivi di sicurezza, i moduli non possono comunicare quando una web page e’ accessibile tramite protocollo file://. Per usare i moduli, devi accedere alla pagina tramite un web server, ad esempio usando un development server come live-server.

Tuttavia, nella maggior parte dei casi non userai i moduli direttamente nel browser. Invece, i moduli verranno caricati e compilati tramite un build tool esterno come webpack. In queste situazioni userai il module loader fornito dal runtime JavaScript di Node. Questo loader introduce alcune particolarita’ di sintassi, che sono indicate nella spiegazione della sintassi.

Questa sezione descrive gli ES6 modules, usati nelle web applications (in quanto parte della specifica ECMAScript). Tuttavia, Node.js usa di default un sistema alternativo di caricamento dei moduli chiamato CommonJS. Questo usa il metodo built-in require() per caricare un module (che restituisce una singola variable “exported”). I values vengono esportati da un module assegnandoli al global module.exports. Questo corso usera’ esclusivamente ES6 Modules—non usare require()—ma e’ bene conoscere l’approccio alternativo CommonJS quando cerchi aiuto.

Module Syntax

Come in Java o Python, un module JavaScript puo’ “caricare” moduli o librerie esterne usando la keyword import:

//Esempio Java: importa la variable `Random` dal modulo `java.util`
import java.util.Random;
# Esempio Python: importa la variable `randint` dal modulo `random`
from random import randint
//JavaScript:
//importa la variable `Random` da un modulo `util.js`
import { Random } from './util.js';

//importa le variables `randint` e `randrange` da un modulo `random.js`
import { randint, randrange } from './random.js'

Questa e’ la versione piu’ comune della sintassi import di ES6 (chiamata named import): scrivi la keyword import, seguita da un set di braces { } contenente una sequenza di variables separate da virgola che vuoi “importare” da un module specifico. Segue la keyword from, seguita da una string che contiene il relative path allo script del module da importare. Nota che nei sistemi basati su Node (come l’ambiente di build React) includere l’estensione del module e’ opzionale (di default il module loader cerca file che finiscono in .js).

  • Assicurati di includere ./ per indicare che stai caricando un file da una directory specifica. Se lo ometti, il module loader di Node cerchera’ un module installato nel load path (ad es. in node_modules/). Ommetti ./ quando carichi librerie di terze parti come jQuery o React:

    //con jquery installato in `node_modules/`
//importa le variables `$` e `jQuery`
import { $, jQuery } from 'jquery'; //no `./` in front of library name

Nota che le variables importate da un module possono essere di qualsiasi tipo—strings, objects e, piu’ spesso, functions.

Qualsiasi variable che un module import deve essere dichiarata come disponibile nel module da cui proviene—simile all’uso della keyword public in Java. Per farlo, exporti la variable (cosi’ puo’ essere “imported” altrove) usando la keyword export, posta davanti alla dichiarazione della variable:

/*** my-module.js ***/

//definisci ed esporta variables per usarle altrove
export const question = "Why'd the chicken cross the road?";
export const answer = "To get to the other side";

export function laugh() { //una function esportata
    console.log("hahaha");
}

Questo si chiama named export (in contrasto con il default export descritto sotto). Una volta che le variables sono state exportate, possono essere importate da un altro file di script:

/*** index.js ***/

//importa tutte e 3 le variables da `my-modules.js`
//ora tutte e 3 le variables esistono in questo modulo e possono essere usate
import { question, answer, laugh } from './my-module.js';

console.log(question); //=> "Perche' il pollo ha attraversato la strada?"
console.log(answer); //=> "Per arrivare dall'altra parte"
laugh(); //"hahaha"

Importante: solo le variables che export sono rese disponibili agli altri moduli! Qualsiasi cosa che non ha quella keyword non sara’ disponibile—rimarra’ “private” al suo module. La best practice e’ exportare solo i values che sai serviranno ad altri moduli.

Quella sopra e’ la sintassi di base per usare named imports ed exports. Ma ci sono altre opzioni sintattiche quando usi import e export.

Per esempio, puoi usare la keyword as per “aliasare” un value sia quando viene exportato (cosi’ e’ condiviso con un nome diverso) sia quando viene importato (cosi’ viene caricato e assegnato con un nome diverso). Questo e’ particolarmente utile quando vuoi produrre una API pulita per un module (cosi’ exporti values con nomi consistenti, anche se internamente usi altri nomi), o quando vuoi importare una variable con un nome molto lungo.

/*** my-module.js ***/
export function foo() { return 'foo'; } //export nominato standard

//fornisci un "alias" (nome consumer) per il value quando esporti
export { bar as yourBar };

//non sara' disponibile (una function "private")
function baz() { return 'baz'; }
/*** index.js ***/

//fornisci un "alias" per il value quando importi!
import {foo as myFoo} from './my-module.js';
myFoo(); //=> 'foo'
foo(); //errore, non esiste (la variable e' aliasata come `myFoo`)

import { yourBar } from './my-module.js'; //importa il value col nome exportato
yourBar() //=> 'bar'

import { bar } from './my-module.js'; //errore, nessun value `bar` esportato

E’ possibile importare tutto cio’ che e’ stato exportato da un module usando la sintassi import * as. Specifichi un nome per l’object che rappresentera’ i values exportati dal module (ad es. theModule nell’esempio sotto), e ogni variable exportata sara’ una property di quell’object. Questo puo’ essere utile quando potresti aggiungere altri exports a un module durante lo sviluppo, ma non vuoi continuare a modificare l’import statement; d’altro canto, il resto del codice puo’ essere piu’ difficile da leggere perche’ devi accedere alle properties dell’object.

/*** index.js ***/
import * as theModule from './my-module'; //importa tutto cio' che e' stato exportato
                                          //carica come un singolo object con i values
                                          //come properties
theModule.foo(); //=> 'foo'
theModule.yourBar(); //=> 'bar'
theModule.baz(); //Errore (function privata, non exportata)

Infine, ogni module puo’ anche exportare una singola (una sola!) variable default, che fornisce una scorciatoia quando importi quella variable dal module. Specifichi il default export includendo la keyword default subito dopo export:

/*** my-module.js ***/
//questa function e' l'export "default"
export default function sayHello() {
    return 'Ciao mondo!';
}
/*** index.js ***/
import greet from './my-module'; //importa il value di default
                                 //`greet` e' l'alias definito in questo modulo

greet(); //=> "Ciao mondo!"

Quando importi un default export, non includi {} con il nome della variable, ma fornisci invece un nome di variable (“alias”) con cui vuoi riferirti a quel value exportato.

Nota che e’ anche possibile rendere values anonimi default exports:

/*** animals.js ***/
export default ['lion', 'tiger', 'bear']; //export di array anonimo

La tecnica del default export e’ particolarmente comune in framework object-oriented come React, o in framework grandi come Bootstrap con molti moduli indipendenti. Usare default exports ti permette di far si’ che ogni module JavaScript contenga il codice per una singola function o class; quella function diventa il default export, permettendo ad altri moduli di importarla rapidamente e facilmente, ad es. import MyComponent from './MyComponent.js'.

Detto questo, in generale ti consiglio di usare named imports nella maggior parte delle situazioni—c’e’ meno sintassi da ricordare e riduce il numero di decisioni su cosa sia default o no. Devi conoscere i default exports perche’ molte librerie di terze parti li usano, ma per il tuo lavoro e’ spesso meglio restare sui named imports.

15.3 Classes

Le classes non sono piu’ comunemente usate in React e sono un po’ sconsigliate in JavaScript. Questa sezione e’ mantenuta per motivi storici e informativi, ma puo’ essere considerata “deprecated” per questo corso.

Anche se JavaScript e’ principalmente un linguaggio scripting e funzionale, supporta una forma di Object Oriented Programming simile a quella del linguaggio Java. Cioe’, possiamo definire classes di data e metodi che agiscono su quei data, e poi instantiate quelle classi in objects che possono essere manipolati. ES6 introduce una nuova sintassi class per fare in modo che creare classi in JavaScript sembri come creare classi in Java!

Why Classes?

Il punto di usare le classes in programmazione—sia in Java che in JavaScript—e’ fare abstraction: vogliamo encapsulate (“raggruppare”) parti del codice in modo da poter ragionare a un livello piu’ alto. Quindi, invece di pensare al programma solo in termini di Numbers, Strings e Arrays, possiamo pensarlo in termini di Dogs, Cats o Persons.

In particolare, le classes encapsulate due cose:

  1. I data (variables) che descrivono la cosa. Questi sono chiamati attributes, fields o instance variables (variables che appartengono a una specifica instance, ad esempio, della class). Per esempio, potremmo parlare di un object’s Person con name (una String), age (un Number) e Halloween haul (un array di candy).

  2. I behaviors (functions) che operano su, usano o cambiano quei data. Questi sono chiamati methods (tecnicamente instance methods, perche’ operano su una specifica instance della class). Per esempio, una Person potrebbe sayHello(), trickOrTreat() o eatCandy().

In JavaScript, le properties di un Object possono essere viste come gli attributes di quell’object. Per esempio:

const person = {
   name: 'Ada',
   age: 21,
   costume: 'Cheshire Cat'
   trickOrTreat: function(newCandy){
      this.candy.push(newCandy);
   }
}

//parlami di questa persona!
console.log(person.name + " e' un " + person.costume);

Questo Object rappresenta una cosa con attributes name, age e costume (ma non abbiamo ancora indicato che questo Object ha la classification di “Person”). Il value della property trickOrTreat() e’ una function (ricorda: le functions sono values!), ed e’ un esempio di come un Object possa “avere” una function.

  • JavaScript usa persino la keyword this per riferirsi a quale object quella function viene chiamata, proprio come Java! Vedi sotto per maggiori dettagli sulla keyword this e le sue particolarita’.

Una Class (classification) agisce come template/recipe/blueprint per gli objects individuali. Definisce quali data (attributes) e behaviors (methods) hanno. Un object e’ una “instance of” (esempio di) una class: noi instantiate un object a partire da una class. Questo ti permette di creare facilmente piu’ objects, ognuno dei quali puo’ tracciare e modificare i propri data. Le ES6 classes forniscono una sintassi con cui definire questi “template”.

Lo styling preferito di React non si basa piu’ molto sulla class syntax, ma e’ comunque una buona idea avere familiarita’ con essa per aiutare la comprensione e nel caso ti serva per lavori piu’ complessi.

Review: Classes in Java

Per prima cosa, considera la seguente class semplice definita in Java (che dovrebbe essere un ripasso dai corsi di programmazione precedenti):

//dichiarazione di class
public class Person {

    //attributi (private)
    private String firstName;
    private int age;

    //un metodo Constructor
    //viene chiamato quando la class e' istanziata (con `new`)
    //e ha il compito di inizializzare gli attributes
    public Person(String newName, int newAge){
        //assegna i parametri agli attributes
        this.firstName = newName;
        this.age = newAge;
    }

    //restituisce il nome di questa Person
    public String getName() {
        return this.firstName; //restituisce il proprio attribute
    }

    //cresci di un anno
    public void haveBirthday() {
        this.age++; //aumenta l'eta' di questa Person (accedendo al proprio attribute)
    }

    //un metodo che prende una Person come parametro
    public void sayHello(Person otherPerson) {
        //chiama un metodo sull'object parametro per stampare
        System.out.println("Ciao, " + otherPerson.getName());

        //accedi al proprio attribute per stampare
        System.out.println("Ho " + this.age +  " anni");
    }
}

Naturalmente puoi usare questa class (instantiarla e chiamarne i methods) in questo modo:

public static void main(String[] args) {
    //istanzia due nuovi objects Person
    Person aliceObj = new Person("Alice", 21);
    Person bobObj = new Person("Bob", 42);

    //chiama il metodo su Alice (cambiando i suoi fields)
    aliceObj.haveBirthday();

    //chiama il metodo SU Alice, e PASSA Bob come parametro
    aliceObj.sayHello(bobObj);
}

Alcune cose da notare su questa sintassi:

  1. Dichiari (annunci) che stai definendo una class usando la keyword class.
  2. Gli attributes Java sono dichiarati in cima al block della class (ma assegnati nel constructor).
  3. Le classes hanno methods constructor usati per instanziare gli attributes.
  4. I class methods sono dichiarati dentro la dichiarazione della class (dentro il block, indentati di un livello).
  5. I class methods possono accedere (usare) gli attribute variables dell’object riferendosi a this.attributeName.
  6. Tu instantiate objects del tipo della class usando la keyword new e poi chiamando un metodo con il nome della class (ad es. new Person()). Quel metodo e’ il constructor, quindi riceve i parametri del constructor.
  7. Chiami i methods sugli objects usando la dot notation (ad es. object.methodName()).
  8. Gli objects instanziati sono solo variables, quindi possono essere passati ad altri methods.

ES6 Class Syntax

Ecco come creeresti la stessa identica class in JavaScript usando la sintassi ES6:

//dichiarazione di class
class Person {

    //un metodo Constructor
    //viene chiamato quando la class e' istanziata (con `new`)
    //e ha il compito di inizializzare gli attributes
    constructor(newName, newAge) {
        //assegna i parametri agli attributes
        this.firstName = newName;
        this.age = newAge;
    }

    //restituisce il nome di questa Person
    getName() {
        return this.firstName; //restituisce il proprio attribute
    }

    //cresci di un anno
    haveBirthday() {
        this.age++; //aumenta l'eta' di questa Person (accedendo al proprio attribute)
    }

    //un metodo che prende una Person come parametro
    sayHello(otherPerson) {
        //chiama un metodo sull'object parametro per stampare
        console.log("Ciao, " + otherPerson.getName());

        //accedi al proprio attribute per stampare
        console.log("Ho " + this.age +  " anni");
    }
}

Ed ecco come useresti questa class:

//istanzia due nuovi objects Person
const aliceObj = new Person("Alice", 21);
const bobObj = new Person("Bob", 42);

//chiama il metodo su Alice (cambiando i suoi attributes)
aliceObj.haveBirthday();

//chiama il metodo SU Alice, e PASSA Bob come parametro
aliceObj.sayHello(bobObj);

Come puoi vedere, questa sintassi e’ molto, molto simile a Java! Proprio come per le functions JavaScript, la maggior parte dei cambiamenti riguarda l’eliminazione delle dichiarazioni di tipo. In effetti, puoi scrivere una class in Java e poi cancellare alcune parole per ottenere una class ES6.

Cose da notare:

  1. Proprio come in Java, le classes JavaScript si dichiarano usando la keyword class (introdotta in ES6).

    Dai sempre alle classes nomi in PascalCase (iniziando con una lettera maiuscola)!

  2. Le classes JavaScript non dichiarano gli attributes in anticipo (in cima alla class). A differenza di Java, le variables JavaScript esistono sempre, sono solo undefined finche’ non vengono assegnate, quindi non serve dichiararle esplicitamente.

    • In JavaScript, niente e’ private; hai di fatto accesso public a tutti gli attributes e functions.

  3. Le classes JavaScript hanno sempre un solo constructor (se presente), e la function si chiama semplicemente constructor().

    • Questo e’ ancora piu’ chiaro che in Java, dove sai che e’ un constructor solo perche’ manca un return type.

  4. Proprio come in Java, i class methods JavaScript sono dichiarati dentro la dichiarazione della class (dentro il block, indentati di un livello).

    Nota pero’ che non serve usare la parola function per indicare che un method e’ una function; basta fornire il nome e i parametri. Questo perche’ le uniche cose dentro una class sono functions, quindi dichiararlo sarebbe ridondante.

  5. Proprio come in Java, i class methods JavaScript possono accedere (usare) le attribute variables dell’object riferendosi a this.attributeName.

  6. Proprio come in Java, instantiate gli objects del tipo della class usando la keyword new e poi chiamando un method con il nome della class (ad es. new Person()). Quel method e’ il constructor(), quindi riceve i parametri del constructor.

  7. Proprio come in Java, chiami i methods sugli objects usando la dot notation (ad es. object.methodName()).

  8. Proprio come in Java, gli objects instanziati sono solo variables, quindi possono essere passati ad altri methods.

Quindi, in pratica, e’ come Java—tranne per le differenze nel modo di dichiarare le functions e per il fatto che usiamo la parola constructor per i constructor methods.

L’altra differenza e’ che mentre in Java di solito definiamo ogni class nel suo file, in JavaScript spesso crei piu’ classi in un singolo file, allo stesso livello globale in cui dichiari anche le functions non-class:

//script.js
'use strict';

//declare a class
class Dog {
    bark() { /*...*/ }
}

//declare another class
class Cat {
    meow() { /*...*/ }
}

//dichiara una function (non-class)
function petAnimal(animal) { /*...*/ }

//a livello "main", istanzia le class e chiama le functions
const fido = new Dog();
petAnimal(fido); //passa questo object Dog alla function

Anche se la sintassi sopra sembra Java, e’ importante ricordare che le instance di class JavaScript sono ancora semplici Objects come gli altri. Per esempio, puoi aggiungere nuove properties e functions a quell’object, o sovrascrivere il value di qualsiasi property. Anche se sembra una class Java, in realta’ non si comporta come tale.

Inheritance

La sintassi class di ES6 permette anche di specificare class inheritance, con cui una class puo’ extendere un’altra. L’inheritance ti permette di specificare che una class e’ una versione piu’ specializzata di un’altra: cioe’ una versione con “extra abilities” (come methods aggiuntivi).

Come in Java, usi la keyword extends per indicare che una class deve inherit da un’altra:

//La class "parent/super"
class Dog {
  constructor(name) {
      this.name = name;
  }

  sit() {
      console.log('Il cane '+this.name+' si siede. Bravo!');
  }

  bark() {
      console.log('"Bau!"');
  }
}

//La class "child/sub" (eredita abilita' da Dog)
class Husky extends Dog {
    constructor(name, distance) {
        super(name); //chiama il constructor del parent
        this.distance = distance;
    }

    //un nuovo method ("abilita' speciale")
    throwFootball() {
        console.log('Husky '+this.name+' lancia '+this.dist+' yard');
    }

    //override (sostituisce) il method del parent
    bark() {
      super.bark(); //chiama il method del parent
      console.log("(Forza husky!)");
    }
}

//uso
const dog = new Husky("Harry", 60); //crea un Husky
dog.sit(); //chiama il method ereditato
dog.throwFootball(); //chiama il method proprio
dog.bark(); //chiama il method proprio (override)

In questo caso, la class Husky e’ una versione specializzata della class Dog: e’ un Dog con alcune abilita’ speciali (ad es. puo’ lanciare un football). Ci riferiamo alla class base, meno specializzata (Dog), come parent o super class, e alla class derivata, piu’ specializzata (Husky), come child o sub-class.

La sub-class Husky inherits i methods definiti nel parent: anche se la class Husky non ha definito un method sit(), lo ha comunque perche’ il parent lo ha definito! Estendendo una class esistente, ottieni molti methods gratis!

La class Husky puo’ anche override i methods del parent, definendo una versione specializzata (ad es. bark()). Questo e’ utile per aggiungere personalizzazione, o per fornire implementazioni specifiche di callback che possono essere usate da un framework—un pattern che vedrai in React.

Nota che, nonostante questa discussione, JavaScript non e’ in realta’ un linguaggio object-oriented. JavaScript usa invece un prototype system per definire tipi di Objects, che permette la cosiddetta prototypical inheritance. La keyword class di ES6 non cambia questo: e’ semplicemente una “shortcut syntax” per specificare i prototype Objects nello stesso modo supportato fin dalla prima versione di JavaScript. La keyword class rende facile definire qualcosa che sembra e si comporta come una OOP class, ma JavaScript non e’ object-oriented! Vedi questa spiegazione (dettagliata) per ulteriori dettagli.

Working with this

In JavaScript, le functions sono chiamate su Objects usando la dot notation (ad es. myObject.myFunction()). Dentro una function, puoi riferirti all’Object su cui e’ stata chiamata usando la keyword this. this e’ una variable locale che viene implicitamente assegnata all’Object come value.

const doggy = {
  name: "Fido",
  bark: function() {
      console.log(this.name, "abbaia"); //`this` e' l'object su cui e' stata chiamata la function
  }
}

doggy.bark(); //=> "Fido abbaia"
  • Qui this e’ assegnato all’object doggy, su cui e’ stato chiamato .bark().

Ma dato che le functions sono values e possono essere assegnate a piu’ variables (con etichette diverse), l’object su cui una function viene chiamata non e’ necessariamente l’object a cui era stata assegnata inizialmente come property. this si riferisce all’object su cui la function e’ chiamata al momento dell’esecuzione, non al momento della definizione:

//Un object che rappresenta un Dog
const doggy = {
  name: "Fido",
  bark: function() { console.log(this.name + " abbaia"); }
}

// Un object che rappresenta un altro Dog
const doggo = {
  name: "Spot",
  bark: function() { console.log(this.name + " guaisce")}
}

//Questo e' Fido che abbaia
doggy.bark( /*this = doggy*/ ); //=> "Fido abbaia"

//Questo e' Spot che abbaia
doggo.bark( /*this = doggo*/ ); //=> "Spot guaisce"

//Questo e' Fido che usa l'abbaio di Spot!
doggy.bark = doggo.bark; //assegna il value della function a `doggy`
doggy.bark( /*this = doggy*/) //=> "Fido guaisce"
  • Nota come la variable this viene implicitamente assegnata al value dell’object su cui e’ stata chiamata—anche se la function viene assegnata a un nuovo object in seguito!

Ma dato che la variable this si riferisce all’object su cui la function e’ chiamata, possono sorgere problemi per le anonymous callback functions che non sono chiamate su un object in particolare:

class Person {
   constructor(name){ this.name = name; } //constructor base

   //saluta ogni person nell'array dato
   greetAll(peopleArray) {
      //itera su ogni Person usando una callback
      peopleArray.forEach(function(person) {
         console.log("Ciao "+person.name+", sono "+this.name);
      });
   }
}

In questo esempio, la function greetAll() produrra’ un errore: TypeError: Cannot read property 'name' of undefined. Questo perche’ this viene usato dentro una anonymous callback function (la function(person){...})—e quella callback non viene chiamata su un object specifico (nota l’assenza di dot notation). Dato che la callback anonima non viene eseguita su un object, this viene assegnato al value undefined (e non puoi accedere a undefined.name).

La soluzione a questo problema e’ usare le arrow functions. Una arrow function ha la feature speciale di condividere lo stesso this lessicale del codice circostante: cioe’, this non viene riassegnato a un object (inesistente) quando usato dentro una arrow function:

class Person {
   constructor(name){ this.name = name; }

   greetAll(peopleArray) {
      peopleArray.forEach((person) => { //arrow function (differenza sottile)
         console.log("Ciao "+person.name+", io sono "+this.name); //funziona correttamente!
      });
   }
}

Questa proprieta’ rende le arrow functions preziose quando specifichi callback functions, soprattutto quando entrano in gioco classes e objects. Usa sempre arrow functions per le anonymous callbacks!

In alternativa, e’ possibile associare “permanentemente” un particolare value di this a una function, indipendentemente dall’object su cui quella function viene chiamata. Questo si chiama binding di this, e si fa chiamando il metodo .bind() sulla function e passando il value che vuoi assegnare a this. Il metodo .bind() restituisce una nuova function che ha il value associato; spesso poi prendi questa nuova function e la “ri-assegni” alla vecchia variable della function:

//ri-assegna la function
myFunction = myFunction.bind(thisValue);
Questo e’ un pattern comune in React (e ha alcuni minimi benefici di performance), ma per questo corso dovresti restare sulle arrow functions per pulizia e leggibilita’.

Questo capitolo ha descritto alcune delle feature ES6 piu’ comuni e potenzialmente utili. Tuttavia, ricorda che la maggior parte di queste sono solo “syntactic shortcuts” per behavior e funzionalita’ che puoi gia’ ottenere usando JavaScript in stile ES5. Quindi non devi per forza usare queste feature nel tuo codice—anche se possono essere utili e compariranno spesso nel modo in cui usiamo librerie come React.
