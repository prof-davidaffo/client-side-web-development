Sviluppo Web lato client

Chapter 27 Appendice: dispensa rapida di front end

Questa appendice è una dispensa operativa per costruire un piccolo front end partendo da zero. Non sostituisce il libro completo: ne seleziona i concetti indispensabili e li organizza in un percorso breve, adatto a chi ha poche ore e vuole arrivare a una pagina web chiara, ordinata e funzionante.

L’obiettivo non è memorizzare tutto il linguaggio. È capire che cosa fanno HTML, CSS e JavaScript, come collaborano dentro il browser, e come si costruisce una pagina passo per passo fino a un piccolo progetto finale responsive e interattivo.

27.1 Perché imparare a costruire una pagina web

Quasi tutto quello che guardiamo ogni giorno — notizie, social, mappe, servizi pubblici — vive nel browser. Eppure pochissime persone sanno come è fatto quello che usano.

Costruire anche solo una pagina semplice cambia il modo in cui si guarda il web. Non si vede più uno schermo con testi e immagini: si vede una struttura, un sistema, qualcosa che qualcuno ha deciso pezzo per pezzo. E si capisce che quel qualcuno potrebbe essere chiunque.

Questa dispensa non trasforma nessuno in sviluppatore. Ma insegna a ragionare come chi costruisce, non solo come chi consuma.

27.2 Tre linguaggi, tre compiti

Quando apri un sito nel browser, entrano in gioco tre elementi distinti: HTML organizza il contenuto, CSS decide come appare, JavaScript lo rende interattivo.

Una metafora: HTML è lo scheletro e gli oggetti della stanza, CSS sono i colori, le dimensioni, la disposizione, JavaScript sono le azioni possibili dentro la stanza.

La prima cosa da imparare è tenerli separati. Chi li mescola senza criterio produce codice difficile da leggere e da correggere.

In questo percorso non ci occupiamo di server o database — solo di quello che il browser legge e visualizza, che è già più che sufficiente per cominciare.

27.3 Il progetto della dispensa

Per tutta la dispensa useremo un progetto semplice ma completo: una landing page responsive per un Open Day scolastico, con un header e una navigazione, una sezione introduttiva, le attività presentate come card, una sezione contatti con un form, e una piccola interazione in JavaScript. È abbastanza piccolo da finirlo davvero, e abbastanza completo da insegnare tutto quello che serve.

27.4 Strumenti minimi

Bastano un browser moderno (Chrome o Firefox, hanno ottimi strumenti per sviluppatori), Visual Studio Code, e una cartella di progetto organizzata così:

progetto-front-end/
|-- index.html
|-- css/
|   |-- style.css
|-- js/
|   |-- script.js
|-- img/

Separare i file in cartelle distinte non è solo questione di ordine: rende chiaro al browser dove trovare il CSS, il JavaScript e le immagini, e aiuta a non fare confusione quando il progetto cresce.

27.5 Il browser legge file

Una pagina web è un insieme di file collegati tra loro. index.html contiene il contenuto, css/style.css le regole grafiche, js/script.js il comportamento, la cartella img/ le immagini.

Quando apri index.html, il browser legge il file HTML. Se dentro trova un collegamento a un file CSS, lo carica e lo applica. Se trova un collegamento a un file JavaScript, lo esegue.

Il punto da capire subito è che il browser non indovina niente. Devi dirgli esplicitamente quale contenuto c’è, come deve apparire, quali azioni deve compiere.

27.6 I percorsi dei file

Uno degli errori più comuni all’inizio non riguarda il linguaggio, ma i percorsi. Se il browser non trova un file, quel file non viene caricato — senza avvisi visibili sulla pagina.

Se index.html si trova nella cartella principale del progetto, il CSS si collega con css/style.css, il JavaScript con js/script.js, un’immagine con img/foto.jpg:

<link rel="stylesheet" href="css/style.css">
<script src="js/script.js"></script>
<img src="img/palestra.jpg" alt="Ingresso della palestra">

Se scrivi un percorso sbagliato, il CSS non viene applicato, JavaScript non parte, l’immagine non compare. Quando qualcosa non funziona, la prima cosa da controllare è sempre il nome del file e il percorso.

27.7 HTML: la struttura della pagina

HTML non serve a fare grafica. Serve a dire che cosa rappresenta ogni parte del contenuto: un titolo principale si scrive con h1, un sottotitolo con h2, un paragrafo con p, un link con a, un’immagine con img, una sezione con section.

27.7.1 La struttura base

<!doctype html>
<html lang="it">
  <head>
    <meta charset="utf-8">
    <meta name="viewport" content="width=device-width, initial-scale=1">
    <title>Open Day</title>
    <link rel="stylesheet" href="css/style.css">
  </head>
  <body>
    <header>
      <h1>Open Day 2026</h1>
    </header>

    <main>
      <section>
        <h2>Benvenuti</h2>
        <p>Questa pagina presenta il nostro evento.</p>
      </section>
    </main>

    <script src="js/script.js"></script>
  </body>
</html>

<!doctype html> dice al browser che il documento è HTML5. <html lang="it"> contiene l’intera pagina e indica la lingua. Il <head> contiene informazioni per il browser ma non il contenuto visibile: <meta charset="utf-8"> permette di mostrare correttamente caratteri accentati e simboli, il meta viewport serve per il comportamento corretto su smartphone (ne parliamo più avanti), <title> imposta il titolo della scheda. Il <body> è tutto quello che l’utente vede.

Il tag <script> va messo alla fine del body. Se lo metti in alto, JavaScript esegue prima che il browser abbia caricato il resto della pagina, e non trova gli elementi che cerca.

27.7.2 I tag principali

Per un primo progetto bastano pochi tag, usati con criterio.

h1, h2, h3 sono i titoli in ordine di importanza — in genere si usa un solo h1 per pagina. p è un paragrafo. a crea un collegamento: href="#contatti" porta a una sezione interna della stessa pagina (usando l’id della sezione), href="https://..." porta a un altro sito. img inserisce un’immagine, con src per il percorso e alt per la descrizione testuale — quest’ultimo è importante perché viene letto da chi non vede la pagina, e compare quando l’immagine non si carica.

Per organizzare la pagina esistono tag con significato preciso: header per l’intestazione, main per il contenuto principale, section per un blocco con un tema, footer per la parte finale, nav per la navigazione, article per un contenuto autonomo come una card. Usare questi tag è meglio che riempire tutto di div generici, perché la struttura diventa più leggibile.

27.7.3 Micro-esercizio

Partendo dall’HTML minimo, aggiungi una sezione “Chi siamo”, una sezione “Attività”, e almeno un link interno che punti ai contatti.

27.8 La struttura completa del progetto

<!doctype html>
<html lang="it">
  <head>
    <meta charset="utf-8">
    <meta name="viewport" content="width=device-width, initial-scale=1">
    <title>Open Day</title>
    <link rel="stylesheet" href="css/style.css">
  </head>
  <body>
    <header>
      <h1>Open Day 2026</h1>
      <nav>
        <a href="#chi-siamo">Chi siamo</a>
        <a href="#attivita">Attività</a>
        <a href="#contatti">Contatti</a>
      </nav>
    </header>

    <main>
      <section id="chi-siamo">
        <h2>Chi siamo</h2>
        <p>Presentiamo i laboratori e gli spazi della scuola.</p>
      </section>

      <section id="attivita">
        <h2>Attività</h2>
        <div class="cards">
          <article class="card">
            <h3>Laboratori</h3>
            <p>Visita agli spazi di informatica, chimica e robotica.</p>
          </article>

          <article class="card">
            <h3>Incontri</h3>
            <p>Docenti e studenti raccontano l'esperienza scolastica.</p>
          </article>

          <article class="card">
            <h3>Orientamento</h3>
            <p>Informazioni sui percorsi di studio e sulle iscrizioni.</p>
          </article>
        </div>
      </section>

      <section id="contatti">
        <h2>Contatti</h2>
        <p>Email: <a href="mailto:info@scuola.it">info@scuola.it</a></p>
      </section>
    </main>

    <footer>
      <p>Open Day 2026</p>
    </footer>

    <script src="js/script.js"></script>
  </body>
</html>

In questo HTML compaiono due attributi fondamentali. id identifica un elemento in modo univoco — non ci possono essere due elementi con lo stesso id nella stessa pagina. Serve a creare link interni come <a href="#contatti"> e a selezionare elementi da JavaScript o CSS. class invece raggruppa elementi che devono condividere uno stile: se scrivi la classe card su tre article, puoi definire una sola regola CSS che si applica a tutti e tre. Lo stesso elemento può avere più classi separate da uno spazio: class="card evidenziata".

27.9 CSS: dare forma alla pagina

Apri ora la pagina nel browser: vedrai quasi tutto senza stile, testo nero su sfondo bianco, nessuna spaziatura. È normale. HTML organizza il contenuto, CSS decide come appare.

27.9.1 Come funziona una regola CSS

Una regola CSS ha sempre la stessa forma:

selettore {
  proprietà: valore;
  proprietà: valore;
}

Il selettore dice a quali elementi si applica la regola. section seleziona tutti gli elementi section della pagina. .card (con il punto) seleziona tutti gli elementi con class="card". #contatti (con il cancelletto) seleziona l’elemento con id="contatti". nav a seleziona tutti i link che si trovano dentro un nav — utile quando vuoi applicare uno stile solo in un certo contesto.

27.9.2 Colori e misure

I colori in CSS si scrivono principalmente in tre modi. I nomi (white, black, red) funzionano solo per i colori base. Il formato esadecimale (#1f3c88) è il più usato — sei cifre che codificano rosso, verde e blu; per trovare il codice di qualsiasi colore cerca “hex color picker” online. Il formato rgb(31, 60, 136) esprime gli stessi valori in forma decimale.

Per le misure, i due formati più comuni sono px (pixel fissi, adatti per i bordi) e rem (relativo alla dimensione base del testo del browser, di solito 16px). 1rem equivale a 16px, 2rem a 32px. Per spaziature e font conviene usare rem, perché si adatta se l’utente cambia le impostazioni del browser.

27.9.3 margin e padding

Questa distinzione va capita subito: margin è lo spazio esterno all’elemento, tra lui e gli altri; padding è lo spazio interno, tra il bordo e il contenuto. Se una card è troppo attaccata alle altre, aumenta il margin. Se il testo dentro la card è schiacciato contro il bordo, aumenta il padding.

.card {
  margin: 1rem;      /* spazio fuori dalla card */
  padding: 1.5rem;   /* spazio dentro la card */
}

27.9.4 Il file CSS completo

Crea css/style.css con queste regole:

body {
  margin: 0;
  font-family: Arial, sans-serif;
  line-height: 1.6;
  color: #222222;
  background-color: #f4f4f4;
}

header {
  background-color: #1f3c88;
  color: white;
  padding: 1.5rem 1rem;
}

nav {
  margin-top: 0.75rem;
}

nav a {
  color: white;
  text-decoration: none;
  margin-right: 1.5rem;
  font-weight: bold;
}

nav a:hover {
  text-decoration: underline;
}

main {
  max-width: 900px;
  margin: 0 auto;
  padding: 1rem;
}

section {
  background-color: white;
  margin-bottom: 1.5rem;
  padding: 1.5rem;
  border-radius: 8px;
}

h2 {
  margin-top: 0;
  color: #1f3c88;
}

.card {
  border: 1px solid #dddddd;
  padding: 1.25rem;
  border-radius: 8px;
  background-color: #fafafa;
}

footer {
  text-align: center;
  padding: 1rem;
  font-size: 0.875rem;
  color: #666666;
}

Il selettore nav a:hover introduce le pseudo-classi: :hover si attiva quando il mouse passa sopra l’elemento. È uno dei modi più semplici per aggiungere un feedback visivo senza JavaScript.

27.9.5 Micro-esercizio

Cambia il colore dell’header con un codice hex di tua scelta, modifica il padding di una .card e osserva come cambia lo spazio interno, e aggiungi un bordo sinistro colorato alle sezioni con border-left: 4px solid #1f3c88;.

27.10 Responsive: la pagina su mobile

Una pagina costruita guardando solo lo schermo del computer può rompersi su smartphone: il testo diventa minuscolo, le colonne si sovrappongono, le immagini escono dal bordo.

Il front end moderno segue un’idea precisa: si costruisce prima una versione semplice per schermi piccoli, poi si migliora per schermi più grandi. Questa idea si chiama mobile-first.

Il <meta name="viewport"> che hai già nel <head> è il primo passo: senza di esso, i browser mobile rimpiccioliscono la pagina come se fosse un desktop. Con quel tag, la pagina si adatta alla larghezza reale del dispositivo.

Per le immagini, questa regola evita che escano dai contenitori:

img {
  max-width: 100%;
  display: block;
}

max-width: 100% fa sì che l’immagine non superi mai la larghezza del suo contenitore. display: block rimuove lo spazio bianco che i browser lasciano sotto le immagini inline.

27.10.1 Layout a colonne con Flexbox

Per disporre le card affiancate su schermi larghi si usa Flexbox, un sistema CSS per gestire layout in riga o in colonna:

.cards {
  display: flex;
  flex-direction: column;
  gap: 1rem;
}

@media (min-width: 768px) {
  .cards {
    flex-direction: row;
  }

  .card {
    flex: 1;
  }
}

Di default le card stanno una sotto l’altra (flex-direction: column). Quando lo schermo arriva a 768px di larghezza, entrano in gioco le regole dentro @media (min-width: 768px): le card si dispongono in riga e con flex: 1 si dividono equamente lo spazio disponibile.

Il blocco @media è una media query: una regola CSS condizionale che si attiva solo se il browser soddisfa una certa condizione — in questo caso la larghezza minima dello schermo. È lo strumento principale del design responsive.

27.10.2 Micro-esercizio

Riduci la finestra del browser sotto i 768px (trascinando il bordo, oppure dagli strumenti sviluppatore). Le card devono stare una sotto l’altra. Poi allarga: devono andare affiancate. Se non succede, controlla la media query e la classe cards.

27.11 Il form dei contatti

Aggiungiamo la sezione contatti con un form. Anche senza un server che riceve i dati, costruire un form è utile per esercitarsi con interfacce realistiche.

27.11.1 HTML

<section id="contatti">
  <h2>Contatti</h2>
  <p>Compila il form per ricevere informazioni sull'Open Day.</p>

  <form class="contact-form">
    <label for="nome">Nome</label>
    <input id="nome" type="text" placeholder="Il tuo nome">

    <label for="email">Email</label>
    <input id="email" type="email" placeholder="la.tua@email.it">

    <label for="messaggio">Messaggio</label>
    <textarea id="messaggio" rows="4" placeholder="Scrivi qui la tua domanda"></textarea>

    <button type="submit">Invia</button>
  </form>
</section>

label descrive il campo — l’attributo for deve corrispondere esattamente all’id del campo collegato, così cliccando sull’etichetta il cursore va nel campo. input è un campo a riga singola; type="email" attiva la validazione automatica del formato. textarea è un’area di testo più grande; rows controlla l’altezza iniziale.

27.12 Quando qualcosa non funziona

Nel front end capita spesso che un file non venga caricato, un selettore sia scritto male o JavaScript non trovi l’elemento giusto. Gli strumenti per sviluppatori del browser sono indispensabili. Si aprono con F12 oppure con tasto destro su qualsiasi elemento → “Ispeziona”.

La Console mostra errori e messaggi. Se JavaScript si interrompe, qui compare il motivo con il numero di riga. Cannot read properties of null significa che JavaScript ha cercato un elemento che non esiste nella pagina. Uncaught SyntaxError è un errore di scrittura nel codice. Puoi anche scrivere direttamente nella console per testare piccoli pezzi di codice al volo.

La scheda Elementi mostra l’HTML realmente caricato dal browser, le classi presenti su ogni elemento, e quali regole CSS si stanno applicando — a volte una regola ne sovrascrive un’altra senza che te ne accorga. Puoi modificare temporaneamente i valori CSS direttamente nel browser per fare prove veloci, senza toccare i file.

La scheda Rete mostra se un file è stato caricato correttamente. Se il CSS non si applica o un’immagine non compare, ricarica la pagina con questa scheda aperta: i file con un errore appaiono in rosso con il codice 404, che significa percorso sbagliato.

Quando qualcosa va male, prima di cambiare codice a caso, fai tre controlli: il file è collegato con il percorso corretto? L’id o la class nell’HTML corrispondono a quelli usati in CSS o JavaScript? La console mostra qualche errore?

27.13 JavaScript: aggiungere interazione

Fino a qui la pagina è statica. JavaScript serve a far succedere qualcosa in risposta a un’azione dell’utente.

27.13.1 Il DOM

Quando JavaScript lavora sulla pagina, non ragiona sul file .html come testo: ragiona sugli elementi già caricati dal browser. Questa rappresentazione si chiama DOM (Document Object Model). Il browser trasforma l’HTML in una struttura ad albero di elementi, JavaScript li seleziona e li modifica.

Il metodo più usato è document.querySelector(), che accetta un selettore CSS:

document.querySelector('#contatti')   // l'elemento con id="contatti"
document.querySelector('.card')       // il primo elemento con class="card"
document.querySelector('h1')          // il primo h1 della pagina

27.13.2 Primo script: cambiare il testo di un paragrafo

HTML:

<section id="info">
  <h2>Informazioni</h2>
  <p id="message">Clicca il pulsante per aggiornare il testo.</p>
  <button id="change-text">Aggiorna testo</button>
</section>

JavaScript in js/script.js:

'use strict';

const button = document.querySelector('#change-text');
const message = document.querySelector('#message');

button.addEventListener('click', function () {
  message.textContent = 'Il testo è stato aggiornato con JavaScript.';
});

'use strict' attiva una modalità più rigida che aiuta a evitare errori banali. document.querySelector seleziona gli elementi e li salva in due costanti. addEventListener('click', ...) dice al browser di eseguire la funzione quando il bottone viene cliccato. message.textContent = ... cambia il testo del paragrafo.

Questo è il pattern fondamentale del front end con JavaScript: seleziono un elemento, ascolto un evento, modifico il DOM.

27.13.3 Micro-esercizio

Modifica l’esempio in modo che il bottone mostri messaggi diversi a ogni click. Suggerimento: usa una variabile che tiene il conto di quante volte è stato premuto.

27.13.4 Secondo script: mostrare e nascondere un elemento

Questo pattern ricorre ovunque — menu mobili, FAQ espandibili, pannelli di dettaglio.

HTML:

<button id="toggle-info">Mostra dettagli</button>
<p id="extra-info" class="hidden">
  I dettagli dell'evento saranno comunicati via email agli iscritti.
</p>

CSS:

.hidden {
  display: none;
}

JavaScript:

'use strict';

const toggleButton = document.querySelector('#toggle-info');
const extraInfo = document.querySelector('#extra-info');

toggleButton.addEventListener('click', function () {
  extraInfo.classList.toggle('hidden');
});

classList.toggle('hidden') aggiunge la classe se è assente, la toglie se è presente. JavaScript non cambia direttamente lo stile: aggiunge o toglie una classe, e CSS decide cosa fare con quella classe. JavaScript decide quando cambiare, CSS decide come appare il risultato. Tenere separati questi due compiti rende il codice più ordinato.

27.13.5 Terzo script: evidenziare il link attivo nel menu

Quando l’utente clicca un link del menu, è utile evidenziarlo per mostrare dove si trova nella pagina.

CSS da aggiungere:

nav a.active {
  border-bottom: 2px solid white;
  padding-bottom: 2px;
}

JavaScript:

'use strict';

const navLinks = document.querySelectorAll('nav a');

navLinks.forEach(function (link) {
  link.addEventListener('click', function () {
    navLinks.forEach(function (l) {
      l.classList.remove('active');
    });
    link.classList.add('active');
  });
});

document.querySelectorAll (con la s finale) restituisce tutti i link dentro nav, non solo il primo. .forEach() esegue una funzione per ciascuno di loro. Quando si clicca un link, prima si rimuove la classe active da tutti, poi la si aggiunge solo a quello cliccato.

27.14 Come lavorare

L’ordine giusto è: scrivi prima l’HTML e controlla che il contenuto sia ordinato; aggiungi il CSS per migliorare l’aspetto; verifica che funzioni su schermi diversi; aggiungi JavaScript solo quando HTML e CSS funzionano già.

Se provi a fare tutto insieme, gli errori diventano difficili da isolare e capire.

27.15 Errori comuni

In HTML: dimenticare di chiudere un tag, usare troppi div generici invece dei tag semantici, mettere un h3 prima di un h2, dimenticare alt nelle immagini, collegare i file con percorsi sbagliati.

In CSS: dimenticare il punto e virgola alla fine di una dichiarazione, confondere margin e padding, usare stili inline direttamente nell’HTML invece del file CSS.

In JavaScript: mettere lo script nell’head invece che in fondo al body, cercare un elemento con un id scritto diversamente rispetto all’HTML, non aprire la console per leggere i messaggi di errore.

27.16 Roadmap del progetto

Costruisci il progetto in quattro versioni. Ognuna aggiunge qualcosa alla precedente.

Versione 1 — Struttura. Costruisci la pagina completa: header con titolo e menu, sezione “Chi siamo”, sezione “Attività” con almeno tre card, sezione “Contatti” con un form, footer. L’HTML deve essere ordinato, ben indentato, con i percorsi dei file che funzionano.

Versione 2 — Stile. Aggiungi colori coerenti nell’header e nelle sezioni, spaziature leggibili, stile al menu e alle card, stile al form. La pagina deve essere presentabile anche senza JavaScript.

Versione 3 — Responsive. Le immagini devono restare dentro i contenitori, le card in colonna su mobile e affiancate su schermi larghi, i testi leggibili a ogni larghezza. Usa la media query a 768px.

Versione 4 — Interazione. Aggiungi almeno una interazione utile: un bottone che mostra o nasconde un dettaglio, il link attivo nel menu che si evidenzia, un messaggio di conferma quando si preme Invia.

27.17 Consegna minima

Il progetto finale deve avere una pagina completa e navigabile, almeno tre sezioni ben distinte, uno stile CSS coerente, un layout responsive funzionante, e una interazione JavaScript semplice ma utile.

27.18 Checklist finale

  1. index.html è ordinato e ben indentato.
  2. CSS e JavaScript sono collegati con il percorso corretto.
  3. Le immagini si caricano e hanno alt.
  4. I titoli seguono un ordine logico (h1, h2, h3).
  5. Il testo è leggibile e i colori hanno contrasto sufficiente.
  6. Le sezioni sono visivamente distinte.
  7. La pagina funziona anche su schermo piccolo.
  8. I link interni ed esterni funzionano.
  9. L’interazione JavaScript funziona davvero.
  10. La console del browser non mostra errori evidenti.

27.20 Sintesi

Costruire un front end di base significa imparare una sequenza precisa: organizzare il contenuto con HTML, dargli forma con CSS, aggiungere comportamento con JavaScript.

Se riesci a fare bene queste tre cose in un progetto semplice, hai già costruito una base vera. Non completa, non avanzata — ma vera. Ed è la base su cui si costruisce tutto il resto.
