Accessibilità dei siti web

Chapter 28 Appendice: Dispensa di Front-End Web Development

Questa appendice costituisce una guida operativa all’sviluppo front-end, rivolta a chi si avvicina per la prima volta alla costruzione di pagine web. Il percorso seleziona i concetti fondamentali e li organizza in una progressione logica: dalla struttura del documento alla presentazione grafica, fino al comportamento interattivo.

L’obiettivo non è la copertura esaustiva del linguaggio, bensì la comprensione del ruolo di ciascuna tecnologia — HTML, CSS e JavaScript — e della loro collaborazione all’interno del browser. Al termine del percorso si dispone degli strumenti necessari per costruire una pagina web completa, responsive e interattiva.

28.1 Tecnologie coinvolte

Una pagina web è il risultato della combinazione di tre tecnologie distinte, ciascuna con responsabilità ben delimitate.

HTML (HyperText Markup Language) definisce la struttura semantica del contenuto: stabilisce cosa sia un titolo, un paragrafo, un collegamento, un’immagine o una sezione tematica. HTML non contiene informazioni di presentazione grafica.

CSS (Cascading Style Sheets) controlla l’aspetto visivo degli elementi HTML: colori, tipografia, spaziature, dimensioni e disposizione degli elementi nella pagina.

JavaScript aggiunge comportamento dinamico: consente di modificare il contenuto della pagina in risposta ad azioni dell’utente, senza ricaricare il documento dal server.

La separazione tra questi tre livelli — struttura, presentazione, comportamento — è un principio architetturale fondamentale. Mescolarli senza criterio produce codice difficile da manutenere e da estendere.

Il presente percorso si limita al lato client, ovvero a ciò che il browser legge e visualizza. Argomenti come server, database e linguaggi lato server esulano dall’ambito di questa dispensa.

28.2 Il progetto di riferimento

Per tutta la dispensa si costruisce un unico progetto: una landing page responsive per un Open Day scolastico. Il progetto è composto da un header con navigazione, una sezione introduttiva, un insieme di card per le attività, una sezione contatti con form e un’interazione in JavaScript.

Le dimensioni contenute del progetto lo rendono completabile nel tempo previsto; la varietà degli elementi garantisce di affrontare tutti i concetti essenziali.

28.3 Strumenti necessari

Sono sufficienti:

  • un browser moderno con strumenti per sviluppatori integrati (Google Chrome o Mozilla Firefox);
  • un editor di codice (si consiglia Visual Studio Code).

La struttura di cartelle del progetto è la seguente:

progetto-front-end/
├── index.html
├── css/
│   └── style.css
├── js/
│   └── script.js
└── img/

La suddivisione in sottocartelle non ha scopo meramente estetico: rende esplicita al browser la posizione di ciascuna risorsa e separa responsabilità differenti. Tutti i percorsi usati nei tag HTML devono corrispondere esattamente a questa struttura.

28.4 Come funziona il browser

Una pagina web è un insieme di file collegati. Quando si apre index.html, il browser analizza il documento HTML e costruisce una rappresentazione interna del contenuto. Se il documento contiene riferimenti a file CSS o JavaScript, il browser li recupera e li elabora: le regole CSS vengono applicate agli elementi HTML, il codice JavaScript viene eseguito.

Il browser non esegue inferenze: ogni risorsa deve essere dichiarata esplicitamente e il percorso indicato deve essere corretto. Un percorso errato non produce messaggi visibili sulla pagina, ma impedisce il caricamento della risorsa.

28.4.1 Percorsi relativi

Tutti i file del progetto vengono collegati tramite percorsi relativi rispetto alla posizione di index.html:

<link rel="stylesheet" href="css/style.css">
<script src="js/script.js"></script>
<img src="img/foto.jpg" alt="Descrizione dell'immagine">

In caso di malfunzionamento — stile non applicato, script non eseguito, immagine assente — il primo controllo da effettuare è sempre la correttezza del percorso e del nome del file (incluse maiuscole e minuscole).

28.5 HTML: struttura semantica del documento

HTML descrive il significato del contenuto, non il suo aspetto. Ogni elemento è scelto in base a ciò che rappresenta, non in base a come viene visualizzato di default dal browser.

28.5.1 Struttura base del documento

<!doctype html>
<html lang="it">
  <head>
    <meta charset="utf-8">
    <meta name="viewport" content="width=device-width, initial-scale=1">
    <title>Open Day</title>
    <link rel="stylesheet" href="css/style.css">
  </head>
  <body>
    <header>
      <h1>Open Day 2026</h1>
    </header>

    <main>
      <section>
        <h2>Benvenuti</h2>
        <p>Questa pagina presenta il nostro evento.</p>
      </section>
    </main>

    <script src="js/script.js"></script>
  </body>
</html>

Le componenti di questa struttura hanno ruoli precisi:

  • <!doctype html> dichiara che il documento è conforme allo standard HTML5.
  • <html lang="it"> è l’elemento radice; l’attributo lang indica la lingua principale del documento.
  • <head> contiene metadati non visibili: la codifica dei caratteri (utf-8, necessaria per accentate e simboli), il meta viewport (indispensabile per il comportamento corretto su dispositivi mobili), il titolo della scheda e il collegamento al foglio di stile.
  • <body> contiene tutto il contenuto visibile.
  • Il tag <script> è posizionato alla fine del <body>, prima della chiusura. Se inserito nell’<head>, il codice JavaScript verrebbe eseguito prima che il browser abbia elaborato gli elementi HTML, rendendo impossibile selezionarli.

28.5.2 Elementi semantici principali

Gli elementi di titolazione h1–h6 definiscono una gerarchia: h1 è il titolo principale della pagina (se ne usa uno solo), h2 i titoli di sezione, h3 i sottotitoli. L’ordine deve essere rispettato: saltare livelli — ad esempio passare da h1 a h3 — compromette la chiarezza della struttura e il posizionamento nei motori di ricerca.

Gli elementi strutturali principali sono:

  • header: intestazione della pagina o di una sezione.
  • nav: blocco di navigazione contenente link.
  • main: contenuto principale della pagina; ne esiste uno solo per documento.
  • section: raggruppamento tematico di contenuto, tipicamente con un titolo.
  • article: contenuto autonomo e riutilizzabile, come una card o un articolo.
  • footer: piè di pagina.

L’elemento generico div non ha significato semantico: va usato solo quando nessun elemento semantico risulta appropriato.

L’elemento a crea collegamenti ipertestuali. L’attributo href="#contatti" punta a un elemento con id="contatti" nella stessa pagina; href="https://..." punta a una risorsa esterna. L’elemento img richiede sempre l’attributo alt, che fornisce una descrizione testuale dell’immagine in caso di mancato caricamento.

28.5.3 Attributi id e class

L’attributo id assegna un identificatore univoco a un elemento: non possono esistere due elementi con lo stesso id nello stesso documento. Viene usato per i link interni (<a href="#sezione">) e per selezionare elementi in JavaScript.

L’attributo class assegna uno o più nomi a un elemento, che può condividere con altri. Viene usato principalmente per applicare stili CSS. Uno stesso elemento può avere più classi separate da spazio: class="card evidenziata".

28.5.4 Struttura HTML completa del progetto

<!doctype html>
<html lang="it">
  <head>
    <meta charset="utf-8">
    <meta name="viewport" content="width=device-width, initial-scale=1">
    <title>Open Day</title>
    <link rel="stylesheet" href="css/style.css">
  </head>
  <body>
    <header>
      <h1>Open Day 2026</h1>
      <nav>
        <a href="#chi-siamo">Chi siamo</a>
        <a href="#attivita">Attività</a>
        <a href="#contatti">Contatti</a>
      </nav>
    </header>

    <main>
      <section id="chi-siamo">
        <h2>Chi siamo</h2>
        <p>Presentiamo i laboratori e gli spazi della scuola.</p>
      </section>

      <section id="attivita">
        <h2>Attività</h2>
        <div class="cards">
          <article class="card">
            <h3>Laboratori</h3>
            <p>Visita agli spazi di informatica, chimica e robotica.</p>
          </article>
          <article class="card">
            <h3>Incontri</h3>
            <p>Docenti e studenti raccontano l'esperienza scolastica.</p>
          </article>
          <article class="card">
            <h3>Orientamento</h3>
            <p>Informazioni sui percorsi di studio e sulle iscrizioni.</p>
          </article>
        </div>
      </section>

      <section id="contatti">
        <h2>Contatti</h2>
        <p>Email: <a href="mailto:info@scuola.it">info@scuola.it</a></p>
      </section>
    </main>

    <footer>
      <p>Open Day 2026</p>
    </footer>

    <script src="js/script.js"></script>
  </body>
</html>

Esercizio. Partendo dalla struttura base, aggiungere una sezione “Chi siamo” e una sezione “Attività”. Inserire almeno un link interno che punti alla sezione “Contatti”. Verificare che i valori degli attributi href corrispondano agli id delle sezioni di destinazione.

28.6 CSS: presentazione visiva

Il CSS descrive come gli elementi HTML devono apparire. Non agisce direttamente sul contenuto: stabilisce regole che il browser applica agli elementi corrispondenti.

28.6.1 Sintassi di una regola

Una regola CSS è composta da un selettore e da un blocco di dichiarazioni:

selettore {
  proprietà: valore;
  proprietà: valore;
}

Il selettore determina a quali elementi si applica la regola. I selettori fondamentali sono:

  • Tipo: section — seleziona tutti gli elementi <section> del documento.
  • Classe: .card — seleziona tutti gli elementi con class="card".
  • ID: #contatti — seleziona l’elemento con id="contatti".
  • Discendente: nav a — seleziona i link <a> contenuti dentro un <nav>.

28.6.2 Unità di misura e colori

Per le misure, le unità più usate sono px (pixel, adatta per bordi e spessori fissi) e rem (relativa alla dimensione base del testo del browser, di default 16 px). L’uso di rem per spaziature e dimensioni tipografiche garantisce che il layout si adatti correttamente quando l’utente modifica le preferenze di testo nel browser. Il rapporto di conversione è: 1rem = 16px, 1.5rem = 24px, 2rem = 32px.

I colori si esprimono in tre formati principali:

  • Esadecimale: #1f3c88 — tre coppie di cifre esadecimali che codificano rosso, verde e blu (RGB). È il formato più diffuso nella pratica.
  • RGB decimale: rgb(31, 60, 136) — equivalente al precedente in notazione decimale.
  • Parole chiave: white, black, red — utilizzabili solo per i colori base.

28.6.3 Il modello a scatola: margin e padding

Ogni elemento HTML occupa uno spazio rettangolare composto da contenuto, padding, border e margin. Conoscere la distinzione tra padding e margin è essenziale:

  • padding è lo spazio tra il contenuto dell’elemento e il suo bordo. Aumentarlo allarga visivamente l’area interna dell’elemento.
  • margin è lo spazio tra il bordo dell’elemento e gli elementi circostanti. Determina la distanza tra gli elementi nella pagina.
.card {
  padding: 1.5rem;  /* spazio tra contenuto e bordo */
  margin: 1rem;     /* spazio tra la card e gli elementi adiacenti */
}

28.6.4 Specificità e cascata

Quando più regole CSS si applicano allo stesso elemento e allo stesso attributo, vince quella con specificità maggiore. L’ordine di priorità, dal più basso al più alto, è: selettore di tipo, selettore di classe, selettore di ID, stili inline. In caso di pari specificità, prevale la regola dichiarata per ultima nel foglio di stile.

28.6.5 Foglio di stile completo del progetto

body {
  margin: 0;
  font-family: Arial, sans-serif;
  line-height: 1.6;
  color: #222222;
  background-color: #f4f4f4;
}

header {
  background-color: #1f3c88;
  color: white;
  padding: 1.5rem 1rem;
}

nav {
  margin-top: 0.75rem;
}

nav a {
  color: white;
  text-decoration: none;
  margin-right: 1.5rem;
  font-weight: bold;
}

nav a:hover {
  text-decoration: underline;
}

main {
  max-width: 900px;
  margin: 0 auto;
  padding: 1rem;
}

section {
  background-color: white;
  margin-bottom: 1.5rem;
  padding: 1.5rem;

La dichiarazione margin: 0 auto su main centra orizzontalmente il contenitore quando ha una larghezza massima (max-width) definita: il margine sinistro e destro si dividono equamente lo spazio disponibile.

La pseudo-classe :hover si attiva quando il puntatore si trova sopra l’elemento. È il meccanismo CSS standard per fornire feedback visivo al passaggio del cursore, senza ricorrere a JavaScript.

Esercizio. Modificare il colore di sfondo dell’header con un valore esadecimale diverso. Aggiungere border-left: 4px solid #1f3c88 alle sezioni e osservare l’effetto. Modificare il padding di .card e verificare come cambia lo spazio interno.

28.7 Design responsive

Una pagina progettata esclusivamente per schermi desktop può risultare illeggibile o inutilizzabile su dispositivi mobili: i testi possono essere troppo piccoli, le colonne sovrapporsi, le immagini uscire dal viewport.

Il design responsive è l’insieme di tecniche che consentono a un’unica pagina di adattarsi correttamente a schermi di qualsiasi larghezza.

28.7.1 Il meta viewport

Il tag presente nell’<head>:

<meta name="viewport" content="width=device-width, initial-scale=1">

istruisce il browser mobile a impostare la larghezza del viewport pari alla larghezza fisica del dispositivo, invece di simulare uno schermo desktop rimpicciolito. Senza questo tag, le media query CSS non funzionano correttamente.

28.7.2 Immagini fluide

La regola seguente previene il traboccamento delle immagini dai propri contenitori:

img {
  max-width: 100%;
  display: block;
}

max-width: 100% limita la larghezza dell’immagine a quella del contenitore, qualunque essa sia. display: block elimina lo spazio bianco che i browser inseriscono sotto le immagini trattate come elementi inline.

28.7.3 Flexbox

Flexbox è un modello di layout CSS che distribuisce gli elementi figli di un contenitore lungo un asse (orizzontale o verticale), gestendo automaticamente l’allineamento e la distribuzione dello spazio.

Le proprietà principali del contenitore flessibile sono:

  • display: flex — attiva il modello Flexbox sul contenitore.
  • flex-direction — definisce l’asse principale: row (orizzontale, default) o column (verticale).
  • gap — imposta la distanza uniforme tra gli elementi figli, senza dover ricorrere a margin.
  • justify-content — allineamento degli elementi lungo l’asse principale.
  • align-items — allineamento degli elementi sull’asse trasversale.

La proprietà flex: 1 applicata agli elementi figli li fa espandere in modo uguale per riempire lo spazio disponibile nel contenitore.

28.7.4 Media query

Una media query è una regola CSS condizionale che si applica solo quando determinate condizioni del dispositivo sono soddisfatte. La condizione più comune è la larghezza minima del viewport:

@media (min-width: 768px) {
  /* regole valide solo su schermi larghi almeno 768px */
}

L’approccio mobile-first consiste nel definire prima gli stili per schermi piccoli, poi usare le media query per aggiungere o sovrascrivere le regole al crescere della larghezza. Questo approccio è preferito perché obbliga a ragionare sul contenuto essenziale e produce codice generalmente più ordinato.

28.7.5 Layout responsive delle card

.cards {
  display: flex;
  flex-direction: column;
  gap: 1rem;
}

@media (min-width: 768px) {
  .cards {
    flex-direction: row;
  }

  .card {
    flex: 1;
  }
}

Con questa configurazione, le card si dispongono verticalmente su schermi con larghezza inferiore a 768 px e orizzontalmente su schermi più larghi. La dichiarazione flex: 1 garantisce che ciascuna card occupi una quota uguale dello spazio disponibile.

Esercizio. Ridurre la larghezza della finestra del browser al di sotto di 768 px: le card devono disporsi in colonna. Allargare la finestra: le card devono affiancarsi. Se il comportamento non corrisponde, verificare che la classe cards sia applicata al contenitore nel file HTML e che la media query sia correttamente scritta nel CSS.

28.8 Il form dei contatti

Un form consente all’utente di inserire dati e inviarli. In questo progetto il form non è collegato a un server, ma viene comunque costruito con una struttura vicina a un caso reale.

28.8.1 Struttura HTML del form

<section id="contatti">
  <h2>Contatti</h2>
  <p>Compilare il form per richiedere informazioni sull'Open Day.</p>

  <form class="contact-form">
    <label for="nome">Nome</label>
    <input id="nome" name="nome" type="text" placeholder="Il tuo nome" required>

    <label for="email">Email</label>
    <input id="email" name="email" type="email" placeholder="la.tua@email.it" required>

    <label for="messaggio">Messaggio</label>
    <textarea id="messaggio" name="messaggio" rows="4" placeholder="Scrivi qui la tua domanda" required></textarea>

    <button type="submit">Invia</button>
  </form>
</section>

L’elemento label descrive il campo corrispondente. L’attributo for deve coincidere con l’id del campo a cui si riferisce: questa corrispondenza consente al browser di spostare il focus al campo quando si clicca sull’etichetta, migliorando l’usabilità.

L’elemento input con type="email" attiva la validazione automatica del formato nel browser prima dell’invio. L’elemento textarea produce un’area di testo a più righe; l’attributo rows ne controlla l’altezza iniziale.

28.8.2 id, name e invio del form

In un form reale, i campi hanno quasi sempre sia id sia name, ma i due attributi non svolgono lo stesso ruolo. id identifica l’elemento nella pagina: serve per collegare label e campo, per creare link interni e per selezionare l’elemento in JavaScript. name, invece, identifica il dato inviato dal form: se il browser inviasse il modulo a un server, i valori verrebbero trasmessi usando i nomi dei campi (nome, email, messaggio).

L’attributo booleano required dice al browser che il campo non può essere lasciato vuoto. Se l’utente preme il pulsante di invio senza aver compilato quei campi, il browser blocca il submit e segnala l’errore.

Quando si preme un pulsante type="submit", il browser prova a inviare il form e normalmente ricarica la pagina. In un progetto solo client-side questo comportamento viene spesso intercettato in JavaScript con event.preventDefault(), così si può mostrare un messaggio di conferma senza lasciare la pagina.

28.8.3 Stili CSS del form

.contact-form {
  display: flex;
  flex-direction: column;
  gap: 0.75rem;
  max-width: 500px;
}

.contact-form label {
  font-weight: bold;
  font-size: 0.9rem;
  color: #444444;
}

.contact-form input,
.contact-form textarea {
  padding: 0.6rem 0.75rem;
  border: 1px solid #cccccc;
  border-radius: 6px;
  font-size: 1rem;
  font-family: inherit;
  width: 100%;
  box-sizing: border-box;
}

.contact-form input:focus,
.contact-form textarea:focus {
  outline: none;
  border-color: #1f3c88;
  box-shadow: 0 0 0 3px rgba(31, 60, 136, 0.15);
}

.contact-form button {
  background-color: #1f3c88;
  color: white;
  border: none;
  padding: 0.75rem 1.5rem;
  border-radius: 6px;
  font-size: 1rem;
  cursor: pointer;
  align-self: flex-start;
}

.contact-form button:hover {
  background-color: #162d6b;
}

La dichiarazione box-sizing: border-box modifica il modello di calcolo delle dimensioni: il padding viene incluso nella larghezza totale dichiarata (width: 100%), invece di aggiungersi. Senza questa regola, un campo con width: 100% e padding laterale supererebbe la larghezza del contenitore.

La pseudo-classe :focus si attiva quando un campo è selezionato. L’alone visivo prodotto da box-shadow indica chiaramente all’utente quale campo è attivo, migliorando l’usabilità del form.

28.9 Strumenti per sviluppatori del browser

Gli strumenti per sviluppatori integrati nel browser sono indispensabili per diagnosticare problemi e verificare il comportamento della pagina. Si aprono con il tasto F12 o tramite tasto destro su un elemento → “Ispeziona”.

28.9.1 Pannello Console

La Console registra errori di esecuzione JavaScript e messaggi prodotti dal codice. I messaggi di errore includono la descrizione del problema e il numero di riga.

I messaggi di errore più frequenti sono:

  • Cannot read properties of null: JavaScript ha tentato di accedere a un elemento che non esiste nel documento. Causa tipica: id errato nel selettore o script eseguito prima del caricamento degli elementi.
  • Uncaught SyntaxError: il codice JavaScript contiene un errore di sintassi che ne impedisce l’esecuzione.
  • Failed to load resource: il browser non ha trovato un file collegato. La causa è quasi sempre un percorso errato.

La Console permette anche di eseguire istruzioni JavaScript direttamente, utile per testare selettori o espressioni in modo rapido.

28.9.2 Pannello Elementi

Il pannello Elementi mostra la struttura HTML effettivamente caricata dal browser, le classi applicate a ciascun elemento e le regole CSS attive. Quando una regola è sovrascritta da un’altra di specificità superiore, viene visualizzata con un tratto barrato. È possibile modificare temporaneamente i valori CSS direttamente nel pannello per valutare l’effetto prima di intervenire sui file.

28.9.3 Pannello Rete

Il pannello Rete mostra le richieste HTTP effettuate dalla pagina durante il caricamento. I file non trovati appaiono con il codice di stato 404. Ricaricare la pagina con questo pannello aperto è il metodo più rapido per verificare che tutti i file CSS, JavaScript e le immagini siano stati caricati correttamente.

28.10 JavaScript essenziale

Prima di manipolare la pagina con il DOM, è utile fissare pochi elementi di sintassi del linguaggio.

28.10.1 Variabili e valori

'use strict';

const titolo = 'Open Day 2026';
let numeroIscritti = 120;
let iscrizioniAperte = true;

console.log(titolo);

const si usa per valori che non devono essere riassegnati; let per valori che possono cambiare durante l’esecuzione del programma. JavaScript distingue tra stringhe ('Open Day 2026'), numeri (120) e valori booleani (true, false).

La funzione console.log() stampa un valore nella Console del browser. È il modo più semplice per controllare se una variabile contiene davvero ciò che ci si aspetta.

28.10.2 Condizioni e funzioni

function creaMessaggio(nome) {
  if (nome === '') {
    return 'Benvenuto';
  }

  return 'Benvenuto, ' + nome;
}

console.log(creaMessaggio('Marta'));

Una funzione raccoglie istruzioni sotto un nome riutilizzabile. L’istruzione if permette di prendere decisioni; return restituisce il risultato della funzione. L’operatore === confronta due valori in modo rigoroso ed è quello da preferire nei controlli di uguaglianza.

28.11 JavaScript: comportamento dinamico

JavaScript consente di modificare il contenuto e la struttura della pagina in risposta ad azioni dell’utente, senza ricaricare il documento. Opera su una rappresentazione interna che il browser costruisce a partire dall’HTML.

28.11.1 Il Document Object Model (DOM)

Quando il browser analizza un documento HTML, costruisce una struttura ad albero in memoria chiamata DOM (Document Object Model). Ogni elemento HTML corrisponde a un nodo dell’albero. JavaScript interagisce con la pagina leggendo e modificando questo albero, non il file HTML sorgente.

28.11.2 Selezione degli elementi

Il metodo document.querySelector() restituisce il primo elemento del documento che corrisponde al selettore CSS fornito. Il metodo document.querySelectorAll() restituisce tutti gli elementi corrispondenti.

document.querySelector('#contatti')   // elemento con id="contatti"
document.querySelector('.card')       // primo elemento con class="card"
document.querySelector('h1')          // primo h1 del documento
document.querySelectorAll('nav a')    // tutti i link dentro nav

Se nessun elemento corrisponde al selettore, querySelector restituisce null. Qualsiasi operazione su null produce un errore: è buona pratica verificare che il valore restituito non sia nullo prima di usarlo.

28.11.3 Gestione degli eventi

Il metodo addEventListener registra una funzione da eseguire quando si verifica un evento su un elemento. Il pattern fondamentale è:

elemento.addEventListener('nomeEvento', function () {
  // codice da eseguire
});

Gli eventi principali sono click (clic del mouse o tocco su dispositivi touch), submit (invio di un form), input (modifica del valore di un campo), keydown (pressione di un tasto).

28.11.4 Esempio 1 — Modifica del testo

L’esempio seguente aggiorna il testo di un paragrafo al clic di un pulsante.

HTML:

<section id="info">
  <h2>Informazioni</h2>
  <p id="message">Cliccare il pulsante per aggiornare il testo.</p>
  <button id="change-text">Aggiorna testo</button>
</section>

JavaScript (js/script.js):

'use strict';

const button  = document.querySelector('#change-text');
const message = document.querySelector('#message');

button.addEventListener('click', function () {
  message.textContent = 'Il testo è stato aggiornato.';
});

'use strict' attiva la modalità rigorosa di JavaScript, che produce errori espliciti in presenza di costrutti ambigui. È buona pratica includerla sempre. La proprietà textContent imposta il contenuto testuale dell’elemento, sostituendo quello precedente.

Esercizio. Modificare lo script in modo che il pulsante mostri un messaggio diverso a ogni click. Suggerimento: introdurre una variabile numerica che tenga traccia del numero di click effettuati.

28.11.5 Esempio 2 — Mostrare e nascondere un elemento

Il pattern show/hide ricorre frequentemente nelle interfacce web: menu mobili, sezioni espandibili, pannelli di dettaglio.

HTML:

<button id="toggle-info">Mostra dettagli</button>
<p id="extra-info" class="hidden">
  I dettagli dell'evento saranno comunicati via email agli iscritti.
</p>

CSS:

.hidden {
  display: none;
}

JavaScript:

'use strict';

const toggleButton = document.querySelector('#toggle-info');
const extraInfo    = document.querySelector('#extra-info');

toggleButton.addEventListener('click', function () {
  extraInfo.classList.toggle('hidden');
});

Il metodo classList.toggle('hidden') aggiunge la classe se assente, la rimuove se presente. Si noti la separazione di responsabilità: JavaScript determina quando agire, CSS stabilisce come appare il risultato. Questa divisione rende il codice più leggibile e manutenibile.

28.11.6 Esempio 3 — Link attivo nella navigazione

Evidenziare il link correntemente selezionato nella barra di navigazione fornisce all’utente un riferimento visivo sulla propria posizione nella pagina.

CSS da aggiungere al foglio di stile:

nav a.active {
  border-bottom: 2px solid white;
  padding-bottom: 2px;
}

JavaScript:

'use strict';

const navLinks = document.querySelectorAll('nav a');

navLinks.forEach(function (link) {
  link.addEventListener('click', function () {
    navLinks.forEach(function (l) {
      l.classList.remove('active');
    });
    link.classList.add('active');
  });
});

querySelectorAll restituisce una NodeList, una collezione di elementi su cui è possibile iterare con .forEach(). Al click su un link, la classe active viene rimossa da tutti i link della navigazione, poi aggiunta esclusivamente a quello selezionato.

28.11.7 Esempio 4 — Conferma all’invio del form

In un progetto solo client-side, l’invio del form viene spesso intercettato per evitare il ricaricamento della pagina e mostrare invece un messaggio di conferma.

HTML da aggiungere subito dopo il form:

<p id="form-feedback"></p>

CSS facoltativo:

#form-feedback {
  margin-top: 0.75rem;
  font-weight: bold;
  color: #1f3c88;
}

JavaScript:

'use strict';

const contactForm = document.querySelector('.contact-form');
const feedback    = document.querySelector('#form-feedback');

contactForm.addEventListener('submit', function (event) {
  event.preventDefault();
  feedback.textContent = 'Richiesta inviata correttamente. Vi contatteremo presto.';
  contactForm.reset();
});

Il metodo event.preventDefault() blocca il comportamento standard del browser, che altrimenti invierebbe il form e ricaricherebbe la pagina. La chiamata contactForm.reset() svuota i campi dopo l’invio, riportandoli allo stato iniziale.

28.12 Diagnostica e risoluzione degli errori

Quando la pagina non si comporta come previsto, è utile seguire una procedura sistematica prima di intervenire sul codice.

Verifiche preliminari:

  1. Il file è collegato con il percorso corretto? Controllare il nome del file e della cartella, incluse maiuscole e minuscole.
  2. L’id o la class usati in CSS o JavaScript corrispondono esattamente a quelli dichiarati nell’HTML?
  3. La Console del browser mostra errori? Leggere il messaggio e il numero di riga indicati.
  4. Nel pannello Rete sono presenti richieste con codice 404?

Errori frequenti in HTML: mancata chiusura di un tag; uso di div generico al posto di elementi semantici appropriati; gerarchia dei titoli non rispettata; attributo alt assente nelle immagini; percorsi errati nelle risorse collegate.

Errori frequenti in CSS: punto e virgola mancante alla fine di una dichiarazione; confusione tra margin e padding; selettore che non corrisponde agli elementi HTML (classe o ID scritti diversamente); stili inline nell’HTML invece che nel file CSS.

Errori frequenti in JavaScript: tag <script> posizionato nell’<head> invece che in fondo al <body>; selettore con id che non corrisponde all’HTML; assenza di apertura della Console per la lettura dei messaggi di errore.

28.12.1 Validazione finale

Una pagina può sembrare corretta nel proprio browser e contenere comunque errori di sintassi HTML o CSS. Prima della consegna finale conviene quindi validare il codice con strumenti automatici:

  • W3C HTML Validation Service
  • W3C CSS Validation Service

La validazione non sostituisce i test nel browser, ma aiuta a individuare errori che spesso passano inosservati: tag non chiusi correttamente, attributi invalidi, proprietà CSS scritte male o dichiarazioni fuori standard.

28.13 Ordine di sviluppo consigliato

Lo sviluppo del progetto va affrontato per livelli progressivi, verificando il funzionamento di ciascun livello prima di procedere al successivo.

  1. Struttura HTML. Costruire l’intero documento: header, sezioni, footer. Il file deve essere ben indentato e tutti i percorsi delle risorse devono essere corretti. Aprire la pagina nel browser e verificare che il contenuto compaia.

  2. Presentazione CSS. Applicare colori, tipografia, spaziature e stile agli elementi. La pagina deve essere presentabile anche in assenza di JavaScript.

  3. Layout responsive. Verificare il comportamento su diverse larghezze di viewport. Le card devono disporsi in colonna su schermi stretti e affiancate su schermi larghi.

  4. Interazione JavaScript. Aggiungere i comportamenti dinamici solo dopo che HTML e CSS funzionano correttamente. Testare ogni script aprendo la Console.

  5. Validazione finale. Controllare HTML e CSS con i validator W3C, poi ripetere una verifica completa nel browser.

Affrontare tutti i livelli contemporaneamente rende difficile isolare l’origine degli errori.

28.14 Progetto: specifiche e consegna

28.14.1 Sviluppo per versioni

Versione 1 — Struttura. Pagina completa con header (titolo e navigazione), sezione “Chi siamo”, sezione “Attività” con almeno tre card, sezione “Contatti” con form, footer. Il codice HTML deve essere ordinato e correttamente indentato.

Versione 2 — Stile. Colori coerenti, spaziature leggibili, stile applicato a menu, card e form. La pagina deve essere presentabile senza JavaScript.

Versione 3 — Responsive. Immagini contenute nei propri contenitori; card in colonna su mobile e affiancate su schermi larghi; testi leggibili a ogni larghezza. Media query applicata alla soglia di 768 px.

Versione 4 — Interazione. Almeno un comportamento dinamico funzionante: mostrare/nascondere un elemento, evidenziare il link attivo, visualizzare un messaggio di conferma all’invio del form.

28.14.2 Requisiti minimi

Il progetto consegnato deve includere: una pagina HTML completa e navigabile; almeno tre sezioni distinte; un foglio di stile CSS coerente; un layout responsive funzionante; almeno un’interazione JavaScript operativa.

28.14.3 Checklist finale

  1. index.html è ben strutturato e indentato correttamente.
  2. CSS e JavaScript sono collegati con percorsi corretti e i file esistono nelle posizioni indicate.
  3. Le immagini si caricano e possiedono l’attributo alt.
  4. La gerarchia dei titoli è rispettata (h1 → h2 → h3).
  5. Il testo è leggibile e i colori offrono contrasto sufficiente.
  6. Le sezioni sono visivamente distinte.
  7. La pagina funziona correttamente su viewport stretto (< 768 px).
  8. I link interni puntano agli id corretti.
  9. L’interazione JavaScript funziona e la Console non riporta errori.
  10. HTML e CSS superano la validazione W3C, oppure eventuali avvisi sono stati letti e compresi.
